# 大規模作業ワークフロー

大規模作業を効率的かつ確実に完遂。

## 大規模作業の定義

| 指標 | 閾値 |
|------|------|
| 影響ファイル数 | 10+ |
| 変更行数 | 500+ |
| 推定時間 | 60分+ |
| サブタスク数 | 5+ |

## 自動発動条件

| 規模 | ファイル数 | アクション |
|------|-----------|-----------|
| 小 | 1-2 | 直接実装 |
| 中 | 3-5 | 自動分解 → 順次/並列 |
| 大 | 6+ | Agent階層 + 並列 |

複数レイヤー → レイヤー別分解 / 独立サブタスク → 自動並列。

### 進捗表示フォーマット

```markdown
### Stage 1 [実行中] ████████░░ 80%
| # | タスク | 状態 |
|---|--------|------|
| 1 | タスク名 | 完了/実行中/待機 |
```

## Phase 0: 計画（必須）

### 作業計画書テンプレ

```markdown
## 1. 目的と範囲 — 目的 / 範囲 / 制約
## 2. 影響分析 — ファイル数 / 変更行数 / リスク
## 3. 並列実行計画 — 段階ごとに dev1=タスク1, dev2=タスク2…
## 4. 品質基準 — カバレッジ ≥80% / 型安全性 100%
## 5. 検証計画 — ユニット / 統合 / E2E
## 6. ロールバック計画
```

### リスク評価

| 条件 | リスク | 緩和 |
|------|--------|------|
| ファイル数 >50 | 影響拡大 | 段階的実装、十分なテスト |
| サブタスク >10 | 依存遅延 | 並列最大化 |

## Phase 1: タスク分解

### 分解戦略

| 条件 | 単位 |
|------|------|
| 複数機能 | 機能単位 |
| 複数レイヤー | レイヤー単位 |
| 多数ファイル | ファイル単位 |

### タスクサイズ

| サイズ | 行数 | 対応 |
|--------|------|------|
| 大きすぎ | >200 | 分割 |
| 適正 | 20-200 | そのまま |
| 小さすぎ | <20 | 統合候補 |

### 優先度

- クリティカルパス: +5
- 被依存数 × 2: +N
- 複雑度高（>150行）: +3

## Phase 2: 実行

### 並列実行戦略

1. 依存関係グラフ構築
2. トポロジカルソートでレベル分け
3. 各レベル内で優先度順に並列
4. 全タスク成功 → 次レベル

### チェックポイント（必須）

| タイミング | 内容 |
|-----------|------|
| 25% | テスト実行、型チェック |
| 50% | カバレッジ ≥70%、統合テスト |
| 75% | パフォーマンス、セキュリティスキャン |

## Phase 3: 統合

- **コンフリクト検出**: 同ファイル変更タスク → 順次実行に変更
- **統合テスト**: 失敗 → Developer Agentに差し戻し

## Phase 4: 検証

| 項目 | 内容 |
|------|------|
| 機能要件 | 全仕様充足 |
| パフォーマンス | 応答時間・メモリ基準 |
| セキュリティ | 脆弱性スキャン |
| コード品質 | lint・型チェック |
| ドキュメント | README・API仕様更新 |

### 最終チェックリスト

```
□ 全タスク完了 / テスト Pass / カバレッジ ≥80% / 型安全 100%
□ セキュリティ / パフォーマンス / E2E 完了
□ ドキュメント更新 / コンフリクト解決
```

## Phase 5: 完了レポート

```markdown
# 完了レポート: [プロジェクト名]
## 実行結果 — 総タスク数 / 並列段階数 / 高速化率
## 品質メトリクス — カバレッジ XX% / バグ 0
## 課題と学び — 発生課題 / 次回改善案
```

**計画→分解→実行→統合→検証 の5フェーズで完遂。**
