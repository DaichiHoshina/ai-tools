# 大規模作業ワークフロー(Large-Scale Workflow)

> **目的**: 大規模作業を効率的かつ確実に完遂

---

## 大規模作業の定義

| 指標 | 閾値 |
|------|------|
| 影響ファイル数 | 10個以上 |
| 変更行数 | 500行以上 |
| 推定時間 | 60分以上 |
| サブタスク数 | 5個以上 |

---

## 自動発動条件

| 規模 | 影響ファイル数 | アクション |
|------|-------------|-----------|
| 小規模 | 1-2個 | 直接実装 |
| 中規模 | 3-5個 | 自動分解 → 順次/並列実行 |
| 大規模 | 6個以上 | Agent階層 + 並列実行 |

複数レイヤー変更 → レイヤー別に分解
独立したサブタスク → 自動で並列実行

### 進捗表示フォーマット

## Phase 0: 計画フェーズ(必須)

### 作業計画書

```markdown
## 1. 目的と範囲
- **目的**: [何を達成するか]
- **範囲**: [どこまでやるか]
- **制約**: [制約条件]

## 2. 影響分析
- 影響ファイル数 / 推定変更行数 / リスク

## 3. 並列実行計画
第1段階(並列):
├─ dev1: タスク1
├─ dev2: タスク2
└─ dev3: タスク3

第2段階(並列):
├─ dev1: タスク4
└─ dev2: タスク5

## 4. 品質基準
- テストカバレッジ: ≥ 80% / 型安全性: 100%

## 5. 検証計画
- [ ] ユニットテスト / 統合テスト / E2Eテスト

## 6. ロールバック計画
[問題発生時の対処]
```

### リスク評価基準

| 条件 | リスク | 緩和策 |
|------|--------|--------|
| ファイル数 > 50 | 影響範囲拡大 | 段階的実装、十分なテスト |
| サブタスク数 > 10 | 依存関係遅延 | 並列実行最大化 |

---

## Phase 1: タスク分解フェーズ

### 分解戦略

| 条件 | 分解単位 |
|------|---------|
| 複数機能あり | 機能単位 |
| 複数レイヤーあり | レイヤー単位 |
| 多数ファイルあり | ファイル単位 |

### タスクサイズ基準

| サイズ | 推定行数 | 対応 |
|--------|---------|------|
| 大きすぎ | > 200行 | 分割 |
| 適正 | 20-200行 | そのまま |
| 小さすぎ | < 20行 | 統合候補 |

### 優先度計算

- クリティカルパス上: +5
- 他タスクに依存される数 × 2: +N
- 複雑度高(推定150行超): +3

---

## Phase 2: 実行フェーズ

### 並列実行戦略

1. 依存関係グラフを構築
2. トポロジカルソートでレベル分け
3. 各レベル内で優先度順に並列実行
4. 全タスク成功確認後、次レベルへ

### チェックポイント(必須)

| タイミング | チェック内容 |
|-----------|------------|
| 25%完了 | テスト実行、型チェック |
| 50%完了 | カバレッジ ≥ 70%、統合テスト |
| 75%完了 | パフォーマンステスト、セキュリティスキャン |

---

## Phase 3: 統合フェーズ

### コンフリクト検出

複数タスクが同じファイルを変更する場合 → 順次実行に変更

### 統合テスト実行

- 各コンポーネント組み合わせのテストケースを実行
- 失敗したテストは Developer Agentに差し戻し

---

## Phase 4: 検証フェーズ

### 検証項目

| 項目 | 内容 |
|------|------|
| 機能要件 | 全仕様を満たすか |
| パフォーマンス | 応答時間・メモリ基準 |
| セキュリティ | 脆弱性スキャン |
| コード品質 | lint・型チェック |
| ドキュメント | README・API仕様更新 |

### 最終チェックリスト

```

□ カバレッジ ≥ 80%
□ 型安全性100%
□ セキュリティチェック完了
□ パフォーマンステスト完了
□ 統合テスト完了
□ E2Eテスト完了
□ ドキュメント更新完了
□ チェックポイント全Pass
□ コンフリクト解決済み
□ ロールバック計画準備済み
```
