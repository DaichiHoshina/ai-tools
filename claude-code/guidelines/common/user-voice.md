# user-voice — ヒト向けドキュメント執筆ガイド

`/docs`、`/design-doc`、`/prd`、`/git-push --pr`、`/retrospective`、`/diagnose` 長文出力時に適用。Notion/GitHub/md ファイルに保存され、**他人が読む**ドキュメントを書くための原則。

> **原則**: ドキュメントは記憶ダンプではない。読み手の判断と行動を助ける道具。AI は自分の知識を吐き出すモードで書きがちだが、読み手は記憶を受け取りたいのではなく、決定を下したり作業を進めたい。書くたびに「誰のどの判断を助けるために書いているか」から逆算する。

## 書き出す前に答える4問

白紙に向かう前、またはこのガイドを参照するタイミングで、4問に具体的に答える。抽象的な答えは不可。

1. **読み手は具体的に誰？**
   - 悪い例: 「エンジニア」「ユーザー」
   - 良い例: 「新しく join する senior backend engineer」「非技術職の CEO」「レビュワーの Ayano さん」
2. **読み終えた人に何を決めさせる / やらせる？**
   - 悪い例: 「理解してもらう」「情報提供」
   - 良い例: 「この Design で approve するか決めさせる」「migration 手順を順にコピペ実行させる」
3. **主張を裏付ける数字 or 事例は？**
   - 悪い例: 「大幅に改善」「パフォーマンスが向上」
   - 良い例: 「p99 が 1.2s → 320ms（-73%）」「本番で2回、同じ NullPointerException が発生した」
4. **「なぜ？」と聞かれたら答えは？**
   - 悪い例: 「〜が必須です」「〜を推奨します」（根拠なし）
   - 良い例: 「localStorage に token 置かない。XSS で抜かれるから」

**この4問に答えずに書いた文章は、ほぼ AI 文体になる**。答えた内容を本文に織り込めば、抽象・無根拠・焦点ぼけの大半は自動的に消える。

### Writing Context ブロック（任意の思考補助）

4問の答えを頭で整理するだけだと AI は実際には使わずに書き始めがち。このとき draft 冒頭に以下のコメントブロックを書くと、思考が散らばらず本文に織り込みやすい。**任意**で、必須ではない。

```markdown
<!-- Writing Context（執筆前の4問・書き直しの根拠。最終提出時の削除は任意） -->
<!--
読み手: <具体的な人物像>
読後のアクション: <approve / 実装 / 質問 等>
主要な数字・事例: <測れるものがあれば必須>
なぜこのドキュメントが必要: <問題 or PRD 接続>
-->
```

抽象語（「エンジニア」「ユーザー」「改善」）で埋めても意味がない。具体的な人物像・数字・問題を書く。書いた内容を本文各所に織り込むのが目的。

### 品質検証は review で

生成時は上記の4問・原則5点を参照して書き、書き終えた後は `/review --focus=writing` で機械的に違反を検出する。検出基準は `skills/comprehensive-review/SKILL.md` の writing 観点に定義済み。`/design-doc`、`/prd`、`/docs` は生成フロー末尾で自動実行する構成になっている。

## chat と document の文体を分ける

チャット応答（genshijin モード）と、Notion/GitHub に残るドキュメントは別物。両者を混同すると「箇条書き金太郎飴」になる。

| 場面 | 文体方針 |
|------|---------|
| **Chat 応答**（AI ↔ user 対話） | 体言止め可、助詞省略可、数値で具体化、結論先出し。短く速く返す。 |
| **ヒト向けドキュメント**（Notion/md/PR） | **文として完結（〜する／〜した）**、**主語明示**、**地の文と箇条書きの混在**。読み物として成立させる。 |

箇条書きだけ並べると「何故これを列挙しているか」が消える。箇条書きの前後に **1-3文の地の文** を置き、文脈を繋ぐ。

### 分量の目安

- 1セクションあたり、**地の文 3-5割、箇条書き/表 5-7割** が読みやすい
- 箇条書きは 3-5 項目で区切る。7項目超は表か段落分割を検討
- コードブロックは5行以内、それ以上は意図を1文で補足

## 5つの執筆原則（各 NG / OK 対比）

### 1. 結論先行（TL;DR 冒頭）

Design Doc、PR description、長文レポートは **最初の1-3文で結論**。「本稿では〜について説明する」系の導入は削除する。

| NG | OK |
|----|-----|
| 本ドキュメントでは User Profile API の設計について述べる。API の設計においては複数の観点が〜 | User Profile API を新設する。既存 `/api/users` は認証情報を返すので、顧客向けプロフィール表示には使えない。本稿で設計と移行計画を決める。|

### 2. 言い切りには根拠を併記

「〜が必須」「〜を推奨」を書くなら、**直後または直前に1文で根拠**を置く。読み手が「なぜ？」と聞く前に答える。

| NG | OK |
|----|-----|
| セッショントークンの適切な管理が必須です。 | セッショントークンは cookie `httpOnly` に入れる。localStorage に置くと XSS 経由で盗まれ、token 自体には expiry しかないため全アカウント乗っ取りが可能になる。|
| パフォーマンスの最適化が重要です。 | p99 を 500ms 以内に抑える。これを超えると「遅い」と感じる閾値（RAIL モデル）であり、Stripe の内部調査で離脱率と直接相関していた。|

### 3. 難語は削るより定義併記

専門用語を削ると正確さを失う。**削らず、初出で1文の定義を併記**する。読み手の中に「senior エンジニア」もいれば「PM」もいる想定で書く。

| NG | OK |
|----|-----|
| idempotency を担保する必要があります。 | idempotency（同じリクエストを2回叩いても結果が変わらない性質）を担保する。ネットワーク retry で二重注文が発生しない。|
| Saga パターンで整合性を確保します。 | Saga パターン（複数サービスの更新を、失敗時に補償 transaction で巻き戻す方式）で整合性を確保する。2PC が使えない分散環境向け。|

### 4. 抽象語を数字・事例に置換

「改善」「最適化」「効率化」は **測れないもの**。測った結果の数字、または起きた事例に置換する。

| NG | OK |
|----|-----|
| レスポンス速度を大幅に改善しました。 | p99 が 1.2s → 320ms（-73%）。Redis cache 導入 + N+1 クエリ削除の合算。|
| エラー処理を強化しました。 | 5xx が日次 平均 120件 → 8件。root cause は DB connection timeout の握りつぶしで、`context.WithTimeout` を全 handler に追加した。|
| 運用負荷を軽減します。 | on-call 呼び出しが月 15回 → 3回想定。自動 rollback + health check 追加で、過去4ヶ月のインシデント 18件中 14件が自動復旧可能だった。|

### 5. プロサで繋ぐ（箇条書きの前後に地の文）

箇条書きは便利だが、**何故その列挙なのか・どれが重要か**が消える。前に1-2文の導入、後に1文のまとめ or 強調を置く。

**NG**:
```
## データ分離戦略

- DB 分離
- Schema 分離
- Row 分離
```

**OK**:
```
## データ分離戦略

tenant データの隔離には3つの戦略があり、コストと安全性のトレードオフで選ぶ。既定は Row 分離、compliance 要件（HIPAA/PCI）があれば DB 分離を検討する。

- **DB 分離**: tenant 毎に database 作成。最強の分離、最高のコスト
- **Schema 分離**: 1DB 内に schema 分割。中位のコストだが tenant 数 1000 超で catalog 肥大
- **Row 分離**: 全 table に `tenant_id` 列。最安・最スケール、app 層で誤クロス防御が必要

後から Row → Schema への昇格は tenant 単位で可能。最初から DB 分離はほぼ過剰設計。
```

## 既存の3層チェック（対話型リライト）

上記原則を踏まえたうえで、draft 完成後の仕上げとして以下3層を適用する（`/docs` が主に使用）。

### Layer 1: Intent（意図確認）

**目的**: セクション主旨を言語化させ、主旨外の文を削除候補に。

- **発動条件**: セクション数 ≥ 3 **または** 本文 ≥ 300字
- **質問形式**: 自由記述（assistant が質問して会話停止、ユーザー応答待ち）
- **質問テンプレ**: 「このセクション『<見出し>』で読み手に何を伝えたい？一言で」
- **反映**: 主旨から外れる文を削除候補として提示、ユーザー確認後に削除

### Layer 2: Understanding（理解確認）

**目的**: 難解語・抽象表現をユーザーの言葉で再定義し、AI の表現を置換。

- **対象**: カタカナ6文字以上、3文字以上の英略語、業界固有用語
- **スキップ**: `user_vocabulary.md` の「用語定義」に登録済みの語
- **質問形式**: 自由記述（会話停止、ユーザー応答待ち）
- **質問テンプレ**: 「『<用語>』ここでどういう意味で使った？」
- **反映**:
  1. ユーザー回答文を draft 該当箇所に置換（AI が言い換えない、回答文を原文のまま使う）
  2. 用語と定義を `user_vocabulary.md` の「用語定義」に追記
- **説明できない場合**: 次のアクションを AskUserQuestion で確認 [調べる / 削除 / 曖昧なまま残す]

### Layer 3: Expression（表現確認）

**目的**: AI 定型・硬い文語・**根拠なき評価語** を除去。

- **対象**: NG辞書ヒット（下記）
- **質問形式**: 選択式（`AskUserQuestion` 使用）
- **質問テンプレ**: 「『<語>』AI っぽい or 根拠が必要？」[そのまま / 削除 / 根拠追記 / 書き換え]
- **反映**: 選択に応じて draft を更新、回答を `user_vocabulary.md` に蓄積

## NG辞書

### AI 定型語（削除推奨）

```
効果的に / 効率的に / シームレスに / 直感的に / 革新的な / 素晴らしい / 強力な
より良い / 〜を実現します / 〜を提供します / 〜を可能にします
〜することができます / ご紹介します / ご覧ください / 〜いただけます
ユーザーエクスペリエンス / ユーザビリティ向上
まず〜しましょう / 重要なポイント / 注目すべき点
本機能は / 本ドキュメントは / 本記事では / 本稿では〜について述べる
```

### 硬い文語（柔らかく）

```
〜である / 〜であった / 〜されている / 〜となる
〜に基づき / 〜に関して / 〜について述べる
当該 / 当方 / 貴殿 / 以下に示す
```

### 根拠なき評価語（根拠併記 or 削除）

```
適切な / 最適な / 最適化する / 効率化する / 改善する
重要 / 必須 / 推奨 / 一般的に / 通常 / 基本的に
強化する / 向上させる / 活用する
```

このカテゴリは**削除でなく「根拠を併記」が基本**。使うなら直後に数字か事例、または「なぜそうなのか」の1文を付ける。

## トリアージルール（質問過多回避）

**合計上限 9件**（3+3+3）。

| Layer | 上限 | 省略条件 |
|-------|------|---------|
| 1 Intent | 3件（セクション） | 本文<300字 または セクション<3 |
| 2 Understanding | 3件（難解語） | 既知語 ≥ 3件、または難解語 < 3 |
| 3 Expression | 3件（NG辞書ヒット） | ヒット 0件 |

上位絞り込み: 出現頻度順。既知語は常にスキップ。

## 出力前セルフチェックリスト（必須）

ドキュメントを確定（投稿 / commit / push）する前に、AI が自分で以下を確認する。

```
[ ] TL;DR または結論が冒頭1-3文にある（「本稿では〜」導入は削除済み）
[ ] 「必須」「推奨」「重要」を書いた箇所には、根拠が1文で併記されている
[ ] 初出の専門用語（idempotency、Saga、RLS 等）には定義が併記されている
[ ] 抽象語（改善、最適化、効率化）の代わりに、数字 or 事例がある
[ ] 読み手が次に何をすべきか（approve / 実行 / 質問）が末尾に明示されている
[ ] 冒頭4問（読み手・ゴール・数字・なぜ）に答える内容になっている
```

**合格条件: 6項目中5項目以上**。未達項目は修正してから出力する。これを全 document コマンド（`/docs`、`/design-doc`、`/prd`、`/diagnose` 長文、`/retrospective`、`/git-push --pr`）で共通の合否ルールとする。

## user_vocabulary.md の形式

auto-memory `~/.claude/projects/{project}/memory/user_vocabulary.md` に蓄積。

```markdown
## 用語定義（ユーザーの言葉）
# 形式: 用語 — 定義（YYYY-MM-DD）
- オーケストレーション — 複数エージェントを順番に動かす仕組み（2026-04-21）

## セクション主旨（意図の型）
# 形式: セクション種別 — よく使う主旨表現
- アーキテクチャ — 「何と何がどうつながるか」

## 嫌う表現
# 形式: 元表現 → 処理
- 効果的に → 削除
- ご紹介します → 削除

## 好む言い換え
# 形式: 元表現 → 置換表現
- 実装する → 書く
- 確認する → 見る
```

## コマンドとの接続

### `/docs`（Notion 投稿）

```
Step 5 draft 完成
  ↓
Step 5.5 対話型リライト（本ガイド）
  - 書き出す前の4問は draft 生成時点で使用済み想定
  - Layer 1 → Layer 2 → Layer 3 の順
  - Layer 2 のユーザー回答文は draft にそのまま織り込む（AIで言い換えない）
  - 回答は user_vocabulary.md に追記
  ↓
Step 5.7 出力前セルフチェック（6項目、本ガイド末尾）
  ↓
Step 6 投稿
```

### `/design-doc`（Design Doc md 生成）

書き出し前に4問、draft 完成後にセルフチェック6項目。

### `/prd`（PRD 生成）

同上。要件定義は特に「なぜ？」と「誰のため？」が弱くなりがちなので4問を厳格に通す。

### `/git-push --pr`（PR description 生成）

4問がそのまま PR description の4セクションになる:
- **Why**（何故この変更が必要か、根拠）
- **What changed**（変更内容、数字や事例で）
- **Testing**（どう検証したか）
- **Review focus**（レビュワーに見てほしい箇所、次アクション）

### `/diagnose` 長文 / `/retrospective`

調査レポートや振り返りの長文出力時に、4問と原則1-4を適用。箇条書きだけで終わらせない。

## notion-writing.md との関係

`notion-writing.md` は最終投稿時の Notion 固有フォーマット（主語必須、見出し階層等）。本ガイドはそこに到達する前の **執筆原則とリライト工程**。両者で衝突する場合は `notion-writing.md` を優先。
