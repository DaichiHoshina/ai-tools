# user-voice — ヒト向けドキュメント執筆ガイド

`/docs`、`/design-doc`、`/prd`、`/git-push --pr`、`/retrospective`、`/diagnose` 長文出力時に適用。Notion/GitHub/md ファイルに保存され**他人が読む**ドキュメント原則。

> **原則**: ドキュメントは記憶ダンプではなく、読み手の判断と行動を助ける道具。書くたびに「誰のどの判断を助けるか」から逆算する。

## 書き出す前に答える4問

抽象的な答え不可。

1. **読み手は具体的に誰？** NG「エンジニア」/ OK「新 join の senior backend」「非技術職 CEO」「レビュワー Ayano さん」
2. **読後に何を決めさせる/やらせる？** NG「理解してもらう」/ OK「Design を approve するか決めさせる」「migration をコピペ実行させる」
3. **主張を裏付ける数字 or 事例は？** NG「大幅改善」/ OK「p99 1.2s → 320ms（-73%）」「本番で2回 NPE」
4. **「なぜ？」と聞かれたら？** NG「〜が必須です」/ OK「localStorage に token 置かない、XSS で抜かれる」

**4問に答えずに書いた文章はほぼ AI 文体になる**。答えを本文に織り込めば抽象・無根拠・焦点ぼけは自動消滅。

### Writing Context ブロック（任意）

draft 冒頭に以下コメントブロックを書くと思考が散らばらず織り込みやすい:

```markdown
<!-- Writing Context（4問・最終削除任意）
読み手: <人物像>
読後アクション: <approve / 実装 / 質問 等>
主要な数字・事例: <測れるもの必須>
なぜこれが必要: <問題 or PRD 接続>
-->
```

抽象語で埋めても無意味。具体人物像・数字・問題を書く。

### 品質検証

書き終えた後 `skills/comprehensive-review/skill.md` の writing 観点 NG 表で違反を数え、閾値超なら書き直す。

検査タイミング:
- `/design-doc`: Step 8 で書き出し → Step 8.5 で `Read` + NG 判定 + `Edit` で書き直し loop
- `/prd`: 出力直前に self-review（Phase 4.5）。`--out <path>` 時はファイル経由
- `/docs`: Notion 投稿前（Step 4.8）に draft self-review

合格ライン: Critical 1件以上 or Warning 4件以上で書き直し（最大2 loop）。3 loop 残存はユーザー報告。

## chat と document の文体を分ける

| 場面 | 方針 |
|------|------|
| Chat 応答（genshijin） | 体言止め可、助詞省略可、結論先出し、短く速く |
| ヒト向けドキュメント | **文として完結（〜する／〜した）**、**主語明示**、**地の文と箇条書き混在** |

箇条書きだけだと「何故列挙か」が消える。前後に **1-3文の地の文**。

### 分量目安

- 1セクション: 地の文 3-5割、箇条書き/表 5-7割
- 箇条書きは 3-5 項目。7超は表 or 段落分割
- コードブロック 5行以内、超過は意図1文補足

## 5つの執筆原則

### 1. 結論先行（TL;DR 冒頭）

最初の1-3文で結論。「本稿では〜について述べる」系導入は削除。

| NG | OK |
|----|-----|
| 本ドキュメントでは User Profile API の設計について述べる。〜 | User Profile API を新設する。既存 `/api/users` は認証情報を返すので顧客向け表示には使えない。本稿で設計と移行計画を決める。|

短文 comment / ticket / Slack は **PREP 3点**（結論→理由→次アクション）。詳細は `~/.claude/rules/ai-output.md`。本ガイドの 4問・5原則は**長文ドキュメント** (Notion/Design Doc/PRD/RCA) に適用。

### 2. 言い切りには根拠を併記

「〜が必須」「〜推奨」を書くなら直前後に**1文で根拠**。

| NG | OK |
|----|-----|
| セッショントークンの適切な管理が必須です | セッショントークンは cookie `httpOnly` に。localStorage は XSS 経由で盗まれ、token に expiry しかないため全アカウント乗っ取り可能 |

### 3. 難語は削るより定義併記

専門用語を削ると正確さを失う。**初出で1文の定義を併記**。

| NG | OK |
|----|-----|
| idempotency を担保する必要があります | idempotency（同じリクエストを2回叩いても結果不変）を担保する。retry で二重注文が発生しない |

### 4. 抽象語を数字・事例に置換

「改善」「最適化」「効率化」は測れない。

| NG | OK |
|----|-----|
| レスポンス速度を大幅改善 | p99 1.2s → 320ms（-73%）。Redis cache + N+1 削除 |
| エラー処理を強化 | 5xx 日次 120件 → 8件。`context.WithTimeout` を全 handler に追加 |

### 5. プロサで繋ぐ（箇条書き前後に地の文）

NG: 「DB 分離 / Schema 分離 / Row 分離」だけ並べる。

OK:
```
tenant データ隔離は3戦略あり、コストと安全性のトレードオフで選ぶ。既定は Row 分離、compliance（HIPAA/PCI）あれば DB 分離。

- **DB 分離**: tenant 毎 DB。最強分離・最高コスト
- **Schema 分離**: 1DB 内 schema。中位コスト、tenant 1000+ で catalog 肥大
- **Row 分離**: 全 table に `tenant_id`。最安・最スケール、app 層で誤クロス防御要

Row → Schema 昇格は tenant 単位で可能。最初から DB 分離は過剰設計。
```

## 既存の3層チェック（対話型リライト）

draft 完成後の仕上げ（`/docs` が主に使用）。

| Layer | 目的 | 対象/発動 | 質問形式 |
|-------|------|-----------|---------|
| 1 Intent | セクション主旨を言語化、主旨外を削除候補 | セクション数 ≥3 or 本文 ≥300字 | 自由記述「このセクション『<見出し>』で何を伝えたい？」 |
| 2 Understanding | 難解語をユーザーの言葉で再定義し置換 | カタカナ6文字+/英略3文字+/業界用語、`user_vocabulary.md` 既知語 skip | 自由記述「『<用語>』ここでどういう意味？」 |
| 3 Expression | AI 定型・硬い文語・根拠なき評価語除去 | 下記 NG 辞書ヒット | `AskUserQuestion`「『<語>』AI っぽい or 根拠必要？」[そのまま/削除/根拠追記/書換] |

Layer 2 反映: ユーザー回答文を **AI で言い換えず原文のまま** draft に置換、`user_vocabulary.md`「用語定義」追記。説明できない場合は `AskUserQuestion` で [調べる/削除/曖昧なまま残す]。

## NG辞書

### AI 定型語（削除）

```
効果的に / 効率的に / シームレスに / 直感的に / 革新的な / 素晴らしい / 強力な
より良い / 〜を実現します / 〜を提供します / 〜を可能にします
〜することができます / ご紹介します / ご覧ください / 〜いただけます
ユーザーエクスペリエンス / ユーザビリティ向上
まず〜しましょう / 重要なポイント / 注目すべき点
本機能は / 本ドキュメントは / 本記事では / 本稿では〜について述べる
```

### 硬い文語（柔らかく）

```
〜である / 〜であった / 〜されている / 〜となる
〜に基づき / 〜に関して / 〜について述べる
当該 / 当方 / 貴殿 / 以下に示す
```

### 根拠なき評価語（根拠併記 or 削除）

```
適切な / 最適な / 最適化する / 効率化する / 改善する
重要 / 必須 / 推奨 / 一般的に / 通常 / 基本的に
強化する / 向上させる / 活用する
```

このカテゴリは**削除でなく「根拠併記」が基本**。直後に数字 or 事例 or「なぜ」の1文。

## トリアージルール（質問過多回避）

合計上限 **9件（3+3+3）**。

| Layer | 上限 | 省略条件 |
|-------|------|---------|
| 1 Intent | 3 | 本文<300字 or セクション<3 |
| 2 Understanding | 3 | 既知語 ≥3 or 難解語<3 |
| 3 Expression | 3 | ヒット 0 |

上位絞込: 出現頻度順。既知語は常に skip。

## 出力前セルフチェックリスト（必須）

確定（投稿/commit/push）前に AI 自身が確認。**6項目中5項目以上で合格**。

```
[ ] TL;DR/結論が冒頭1-3文（「本稿では〜」削除済）
[ ] 「必須」「推奨」「重要」に1文の根拠併記
[ ] 初出専門用語に定義併記
[ ] 抽象語（改善/最適化/効率化）の代わりに数字 or 事例
[ ] 読み手の次アクション（approve/実行/質問）が末尾明示
[ ] 4問（読み手/ゴール/数字/なぜ）に答えてる
```

未達項目は修正してから出力。全 document コマンド共通ルール。

## user_vocabulary.md の形式

auto-memory `~/.claude/projects/{project}/memory/user_vocabulary.md` に蓄積。

```markdown
## 用語定義（ユーザーの言葉）
# 形式: 用語 — 定義（YYYY-MM-DD）
- オーケストレーション — 複数エージェントを順番に動かす仕組み（2026-04-21）

## セクション主旨（意図の型）
- アーキテクチャ — 「何と何がどうつながるか」

## 嫌う表現
- 効果的に → 削除
- ご紹介します → 削除

## 好む言い換え
- 実装する → 書く
- 確認する → 見る
```

## Before/After サンプル

| 観点 | Before（AI調） | After（伝わる） |
|------|--------------|----------------|
| 結論先行・根拠 | 本機能は効率的にドキュメント作成を可能にします | Notion へ md を直接投稿できる。手動コピペを無くすのが目的 |
| 難語定義 | CQRS で整合性を担保 | CQRS（読み書きを別モデル分離）で整合性。read は replica から返せて scale 可 |
| 抽象→数字 | エラー処理を強化 | 5xx 日次120件→8件、`context.WithTimeout` 全 handler 追加 |
| プロサ | 箇条書き10項目だけ | 「重要度順、最上位 X 理由〜」の1文 → 3-5項目 → 「残りは参考」の締め |

## コマンドとの接続

| コマンド | 適用 |
|---------|------|
| `/docs` | Step 5 draft → 5.5 対話型リライト（Layer1→2→3）→ 5.7 セルフチェック → 投稿 |
| `/design-doc` | 書き出し前4問、draft 完成後セルフチェック6項目 |
| `/prd` | 同上。「なぜ？」「誰のため？」を厳格に |
| `/git-push --pr` | 4問が PR description 4セクション: **Why** / **What changed**（数字・事例）/ **Testing** / **Review focus** |
| `/diagnose` 長文 / `/retrospective` | 4問と原則1-4 適用、箇条書きだけで終わらせない |

## notion-writing.md との関係

`notion-writing.md` は最終投稿時の Notion 固有フォーマット（主語必須、見出し階層）。本ガイドはそれ以前の**執筆原則とリライト工程**。衝突時は `notion-writing.md` 優先。
