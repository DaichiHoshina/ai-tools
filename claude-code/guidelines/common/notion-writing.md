# Notion ナレッジ管理ガイドライン

## ナレッジベース設計原則

### 階層構造

- **3階層まで**に収める（ワークスペース → カテゴリ → ページ）
- 4階層目が必要なら、ページを分割するサイン
- トップレベルカテゴリは5〜8個が上限（多すぎると迷子になる）

### データベース活用

- 静的ページより**データベース**を優先（検索性・フィルタ性が高い）
- タグはRelationで別データベース化（フリーテキストタグは使わない）
- Linked Databaseで同一情報を複数箇所に表示（複製しない = Single Source of Truth）
- ビューを目的別に分ける（全件一覧 / カテゴリ別 / 最近更新 / 自分が作成）

### 命名規則

| 種別 | 形式 | 例 |
|------|------|-----|
| カテゴリページ | 名詞 | 「認証」「決済」「インフラ」 |
| ナレッジページ | [カテゴリ] 具体内容 | 「[認証] OAuth2.0 フロー設計」 |
| 障害対応 | [YYYY/MM/DD] 障害概要 | 「[2026/04/14] API 504 多発」 |
| ADR | ADR-NNN: 決定事項 | 「ADR-012: セッション管理をRedisに移行」 |

### タグ設計

| タグ種別 | 値の例 | 用途 |
|---------|--------|------|
| カテゴリ | Backend, Frontend, Infra, Design | 技術領域 |
| タイプ | ADR, Runbook, Recipe, Incident, API | ドキュメント種別 |
| ステータス | Draft, Published, Archived | ライフサイクル |
| 重要度 | High, Medium, Low | 優先フィルタ |

## ページ作成ルール

### 構成

- 1ナレッジ=1ページ（複数トピック混ぜない）
- 800〜1,500文字/ページ目安、2,000文字超なら分割
- 冒頭にコールアウトで概要・結論を書く（読者は最初の3行で要否を判断する）

### 見出し・構造

- H2=大見出し、H3=小見出し（H1は使わない）
- H4が必要なら分割のサイン
- 情報量が多い場合は目次をつける

### 視覚要素

- 色は2色まで。赤字は最重要のみ
- テンプレはコードブロック
- トグルは入れ子禁止、短文には使わない
- フロー図はMermaid記法
- 複数列は同カテゴリのみ、左右の情報量を揃える
- 画像は必ずテキストでも完結するよう言語化

### 文体

- 主語必須（「担当者は〜」「ユーザーは〜」）
- 指示語禁止（「これ」「上記」→具体名称で）
- 略語禁止（正式名称を使う）
- 「適宜」「状況に応じて」禁止→IF-THEN形式で明文化

### 表記

- 英数字は半角
- 日付はYYYY/MM/DD
- テーブルはセル結合禁止、空欄は「なし」記入

## テーブル（データベース）設計

### プロパティタイプの使い分け

| データの性質 | 使うプロパティ | 使わないもの | 理由 |
|-------------|--------------|-------------|------|
| カテゴリ・状態 | Select / Status | Text | 表記揺れ防止、フィルタ可能 |
| 複数タグ | Multi-select | Text（カンマ区切り） | 個別フィルタ・集計可能 |
| 日時 | Date | Text | カレンダービュー・ソート可能 |
| 金額・数量 | Number | Text | 合計・平均の自動計算可能 |
| 他テーブル参照 | Relation | Text（名前コピー） | Single Source of Truth |
| 集計値 | Rollup（Relation経由） | 手動入力 | 自動更新 |
| 計算・判定 | Formula | なし | 条件分岐・自動分類に使用 |
| 真偽値 | Checkbox | Select（Yes/No） | フィルタが直感的 |

### プロパティ命名規則

- **具体的な名前**にする（「Info」→「クライアント名」「公開日」）
- 英語の場合はPascalCase（「PublishDate」「ClientBudget」）
- 日本語の場合は短く（「担当者」「期限」「優先度」）
- 同じ意味のプロパティ名をDB間で統一する（「Status」に統一、「状態」「ステータス」混在禁止）

### Select / Multi-select の管理

- 選択肢は**事前定義**する（自由入力させない）
- 表記を統一（「進行中」と「In Progress」を混ぜない）
- 選択肢は10個以内。超えたらグルーピングか別プロパティに分割
- 色は意味を持たせる（赤=緊急、黄=注意、緑=完了 など）

### ビュー設計

| ビュータイプ | 用途 | 例 |
|-------------|------|-----|
| Table | 全件一覧・一括編集 | マスターデータ管理 |
| Board | ステータス別管理 | タスク管理（Todo/Doing/Done） |
| Calendar | 日付ベース管理 | リリーススケジュール |
| List | シンプルな一覧 | ナレッジ一覧 |
| Gallery | カード型表示 | ポートフォリオ・議事録 |
| Timeline | ガントチャート | プロジェクト進行 |

- 1つのDBに**目的別ビューを3〜5個**作る（全件一覧 / 自分の担当 / 今週の期限 等）
- フィルタなしの「Master View」を1つ必ず残す（DB全体構造の把握用）
- ビューごとに表示プロパティを絞る（不要列は非表示に）

### パフォーマンス

- プロパティ数は**15個以内**を目安（多すぎると表示が遅くなる）
- 完了済みレコードは定期的にArchiveビューに移動
- Formulaの入れ子は2階層まで（複雑なら別プロパティに分割）

### アンチパターン

| やりがち | 正しいやり方 |
|---------|------------|
| Textに日付を書く | Date プロパティを使う |
| Textにカテゴリを書く | Select を使う |
| 同じ情報を複数DBに手動コピー | Relation + Rollup で参照 |
| 1つのDBに全種類のデータを詰め込む | 種類別にDBを分けてRelation |
| Select の選択肢に「その他」を入れる | 選択肢を見直すか、Text備考欄を別に用意 |

## 検索性向上

- ページ冒頭に**キーワード列挙**（Notionの検索はページ内全文検索）
- 関連ページへのリンクを「参考」セクションに明記
- 同義語・別名を本文中に含める（「認証」「ログイン」「auth」を全て書く）

## メンテナンス

| 頻度 | アクション |
|------|-----------|
| 作成時 | ステータス=Draft、タグ付け、カテゴリ設定 |
| 公開時 | ステータス=Published、チームに共有 |
| 四半期 | 古いページをレビュー。無効なら Archived に変更 |
| 随時 | 変更があった場合は該当ページを更新（新規作成しない） |

- **更新 > 新規作成**: 同じトピックのページが既にあるなら更新する
- Archived ページは削除しない（検索で過去の経緯を辿れるようにする）
