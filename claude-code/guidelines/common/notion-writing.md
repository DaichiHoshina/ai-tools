# Notion ナレッジ管理ガイドライン

## ナレッジベース設計原則

### 階層構造

- **3階層まで**に収める（ワークスペース → カテゴリ → ページ）
- 4階層目が必要なら、ページを分割するサイン
- トップレベルカテゴリは5〜8個が上限（多すぎると迷子になる）

### データベース活用

- 静的ページより**データベース**を優先（検索性・フィルタ性が高い）
- タグはRelationで別データベース化（フリーテキストタグは使わない）
- Linked Databaseで同一情報を複数箇所に表示（複製しない = Single Source of Truth）
- ビューを目的別に分ける（全件一覧 / カテゴリ別 / 最近更新 / 自分が作成）

### 命名規則

| 種別 | 形式 | 例 |
|------|------|-----|
| カテゴリページ | 名詞 | 「認証」「決済」「インフラ」 |
| ナレッジページ | [カテゴリ] 具体内容 | 「[認証] OAuth2.0 フロー設計」 |
| 障害対応 | [YYYY/MM/DD] 障害概要 | 「[2026/04/14] API 504 多発」 |
| ADR | ADR-NNN: 決定事項 | 「ADR-012: セッション管理をRedisに移行」 |

### タグ設計

| タグ種別 | 値の例 | 用途 |
|---------|--------|------|
| カテゴリ | Backend, Frontend, Infra, Design | 技術領域 |
| タイプ | ADR, Runbook, Recipe, Incident, API | ドキュメント種別 |
| ステータス | Draft, Published, Archived | ライフサイクル |
| 重要度 | High, Medium, Low | 優先フィルタ |

## ページ作成ルール

### 構成

- 1ナレッジ=1ページ（複数トピック混ぜない）
- 800〜1,500文字/ページ目安、2,000文字超なら分割
- 冒頭にコールアウトで概要・結論を書く（読者は最初の3行で要否を判断する）

### 見出し・構造

- H2=大見出し、H3=小見出し（H1は使わない）
- H4が必要なら分割のサイン
- 情報量が多い場合は目次をつける

### 視覚要素

- 色は2色まで。赤字は最重要のみ
- テンプレはコードブロック
- トグルは入れ子禁止、短文には使わない
- フロー図はMermaid記法
- 複数列は同カテゴリのみ、左右の情報量を揃える
- 画像は必ずテキストでも完結するよう言語化

### 文体

- 主語必須（「担当者は〜」「ユーザーは〜」）
- 指示語禁止（「これ」「上記」→具体名称で）
- 略語禁止（正式名称を使う）
- 「適宜」「状況に応じて」禁止→IF-THEN形式で明文化

### 表記

- 英数字は半角
- 日付はYYYY/MM/DD
- テーブルはセル結合禁止、空欄は「なし」記入

## 検索性向上

- ページ冒頭に**キーワード列挙**（Notionの検索はページ内全文検索）
- 関連ページへのリンクを「参考」セクションに明記
- 同義語・別名を本文中に含める（「認証」「ログイン」「auth」を全て書く）

## メンテナンス

| 頻度 | アクション |
|------|-----------|
| 作成時 | ステータス=Draft、タグ付け、カテゴリ設定 |
| 公開時 | ステータス=Published、チームに共有 |
| 四半期 | 古いページをレビュー。無効なら Archived に変更 |
| 随時 | 変更があった場合は該当ページを更新（新規作成しない） |

- **更新 > 新規作成**: 同じトピックのページが既にあるなら更新する
- Archived ページは削除しない（検索で過去の経緯を辿れるようにする）
