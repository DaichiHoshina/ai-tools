# プロンプトベストプラクティス

> **出典**: Boris氏（Claude Code開発者）のヒント#6「プロンプトレベル向上」

---

## まとめ

| 項目 | ポイント |
|------|---------|
| **挑戦させる** | 「最もエレガントな」「創造的な」を明示 |
| **洗練要求** | パフォーマンス・保守性・拡張性を同時に要求 |
| **詳細仕様** | 制約・エッジケース・期待動作を明記 |
| **効果** | 実装品質30%向上、手戻り50%削減（Boris氏） |

**鉄則**: 「動けばいい」ではなく「なぜその設計か」をClaudeに説明させることで、深い理解に基づく実装を引き出す。

---

## 1. Claudeに挑戦させる

「最もエレガントな解決策」「創造的なアプローチ」を明示することで複数案比較+トレードオフ明示が得られる。

**強いプロンプト例（リファクタリング）:**
```
このコードを以下の観点でリファクタリングしてください:
- SOLID原則に準拠
- 技術的負債の削減（重複コード排除）
- テスタビリティの向上（DIパターン活用）

最も効果の高いリファクタリング手法を提案し、
Before/Afterで改善内容を明示してください。
```

---

## 2. 洗練された解決策を要求

「動けばいい」ではなくパフォーマンス・メンテナンス性・拡張性を同時に明示的に要求する。

**強いプロンプト例（API設計）:**
```
商品検索APIを設計してください。
以下を満たす洗練された設計を提案してください:
- パフォーマンス: 100万件データでも1秒以内にレスポンス
- 拡張性: フィルター条件・ソート順の追加が容易
- セキュリティ: SQLインジェクション対策、レート制限
- 運用性: ページング、エラーハンドリング、ログ設計
```

---

## 3. 詳細な仕様を記述

曖昧さを排除し、制約条件・エッジケース・期待動作を明示してClaudeの推測を減らす。

**強いプロンプト例（バリデーション）:**
```
メールアドレスのバリデーションを実装してください。

**仕様**:
- RFC 5322準拠 / 最大長: 254文字
- 許可文字: a-z, A-Z, 0-9, .-_+

**エッジケース**:
- "user+tag@example.com" → 許可
- "user@localhost" → 拒否（TLD必須）
- 連続ドット ".." → 拒否

**エラーメッセージ**:
- 形式不正: "メールアドレスの形式が正しくありません"

テストケース5件以上含めてください。
```

**強いプロンプト例（状態遷移）:**
```
注文のステータス管理を実装してください。

**状態**: DRAFT → PENDING → PAID → SHIPPED / CANCELLED

**遷移ルール**:
- DRAFT → PENDING: items.length > 0 が条件
- PENDING → PAID: 決済成功時のみ
- PAID以降はCANCELLED不可（返金フロー別途）

**不正遷移時**: InvalidStateTransitionError + WARNログ

状態遷移図（Mermaid）も作成してください。
```

---

## 4. アンチパターン

| アンチパターン | 問題 | 改善方針 |
|-------------|------|---------|
| 曖昧な指示（「良い感じに」） | 意図と乖離した実装 | 具体的な基準（変数名・計算量・行数）を明記 |
| 過度に短いプロンプト（「バグ修正して」） | 推測で修正し別バグを生む | 症状・再現手順・期待動作を明記 |
| 制約条件の欠落（「検索機能を追加して」） | パフォーマンス・データ量が不明 | 前提条件・要件・技術選択を明記 |

---

**「何を達成すれば成功か」を具体的に伝えることで、Claudeは最適な手順を自分で組み立てられる。**
