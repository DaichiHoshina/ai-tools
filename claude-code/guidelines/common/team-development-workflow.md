# チーム開発ワークフロー

## 工数見積

### 進め方

1. タスクを細分化して項目を洗い出す
2. DBスキーマ変更がある場合は変更内容・カラムの役割を記載
3. API・画面・バッチ等を洗い出し、それぞれに工数を付ける
4. チーム内共有 → 「考慮漏れ」「実装方針」「日数の妥当性」をレビュー

### 見積フォーマット例

```markdown
## テーブルスキーマの追加・変更

- items テーブル追加
  - id int auto_increment
  - name varchar(255), not null

## API

- GET /items/:id
  - 工数: 1人日
- POST /items
  - 工数: 1人日

## 画面

- /items/:id 詳細ページ - 工数: 1人日
```

### 注意

- 残業時間は勘定に入れない
- 1日を0.8日換算（開発以外の対応を考慮）
- 数行の小タスクはPR上で方針確認すればOK

## PR作成ルール

### PR粒度の目安

| 単位 | 例 |
|------|-----|
| migration単独 | スキーマ変更のみ |
| API 1エンドポイント + テスト | CRUD 1操作 |
| 画面1つ | 詳細ページ、一覧ページ |
| リファクタ単独 | lint修正、コード整理 |

### PR作成時の必須事項

| 項目 | 内容 |
|------|------|
| CIパス | テスト・lint通過後にレビュワーアサイン |
| dev確認 | local/dev環境での動作確認結果を明記 |
| 確認不要の例 | まだ呼ばれないRepository/Usecase実装 |
| レビューポイント | 見てほしい箇所を事前にコメント |
| 関連Issue | `Closes #XXX` で紐づけ |

### ブランチ命名

| パターン | 形式 | 例 |
|---------|------|-----|
| 機能追加 | `{チケット番号}-{概要}` | `5115-order-update` |
| バグ修正 | `hotfix-{概要}` | `hotfix-null-check` |

## レビュープロセス

### レビューコメント添字

| 添字 | 意味 | 対応 |
|------|------|------|
| `must` | 修正必須。リリース不可 | 必ず修正 |
| `imo` | 自分ならこう書く | 修正推奨だが任意 |
| `nits` | 細かい指摘 | 直さなくてもOK |
| `q` | 質問 | 回答のみ |

### レビュー後の流れ

1. 指摘事項を修正
2. Re-request reviewを押す
3. Approve後、PR作成者がマージ
4. main追従ルールは `release-management.md` 参照

### レビュワーの心得

- アサインされた翌日朝イチで開始が理想
- タイミングはレビュワーの裁量

## レビュー頻出パターン

### 設計

| パターン | 説明 |
|---------|------|
| 構造体の使い回し禁止 | 同じフィールド構成でもエンドポイントごとに専用の型を定義。削除時の影響範囲を局所化 |
| 過剰な共通化を避ける | 呼び出し元で分岐済みなら内部で再分岐するヘルパーは不要。SQLの共通化は積極的に |
| 不要なInputを作らない | パラメータがなければ引数なしにする |
| ドメインモデルの変更はメソッドで | フィールド直接書き換え禁止。`SetXxx()` メソッドを実装 |
| API分割・統合の検討 | 新API作成前に既存APIへの機能追加が自然でないか検討 |

### コード品質

| パターン | 説明 |
|---------|------|
| 未使用コード混入禁止 | そのPRで使わないinterface/関数は含めない |
| エラーステータスコード | サーバー側不整合→500、リソース未発見→404。安易に400にしない |
| API呼び出しは並列化 | 複数API呼び出しは `Promise.all` 等で並行実行 |
| ローディング状態管理 | データ取得中に「0件」表示が出ないよう `loading` 状態を管理 |

### migration

| パターン | 説明 |
|---------|------|
| 番号の衝突確認 | mainの最新番号を確認してから振る。マージ直前に再確認 |
| down migrationでDEFAULT明示 | `MODIFY COLUMN` では `DEFAULT NULL` 等を明示 |

### API削除

| クライアント | 削除条件 |
|-------------|---------|
| Webのみ | Web側の参照がなくなれば削除OK |
| ネイティブアプリ | そのAPIを叩くバージョンが強制アップデートで消滅した後に削除 |
