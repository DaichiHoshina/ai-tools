# 品質保証

間違いを防止し成果物の品質を保証する。

## 4原則

| 原則 | 内容 |
|------|------|
| ゼロ欠陥 | 段階的検証・自己レビュー・自動チェック |
| 多層防御 | 作業中→完了時→自動ツール→最終確認（各層 ≥90% 検出） |
| 早期検出 | 作業中=1x、作業後=10x、納品後=100xコスト |
| 継続的改善 | 失敗→原因分析→プロセス改善→再発防止 |

## 作業中の自己質問

```
□ 今やっていることは正しい? 見落としは?
□ エッジケース考慮した? テスト通る?
□ 型安全性は? セキュリティは? パフォーマンスへの影響は?
□ 既存コードを壊してない?
```

## 完了時チェックリスト

### コード品質

```
□ 型安全: any/Any/interface{} 不使用、全型明示
□ テストカバレッジ ≥80%、全エッジケースカバー
□ 関数 ≤50行 / 循環複雑度 ≤10 / 引数 ≤3
□ 意図明確な命名 / マジックナンバー不使用
```

### 機能・非機能要件

```
□ 仕様通り実装、全要件充足、エッジケース考慮、エラーハンドリング適切
□ パフォーマンス: API 応答 <200ms、N+1 なし、メモリリークなし
□ セキュリティ: SQLi/XSS/CSRF 対策、認証・認可実装
□ 可用性: エラー回復、ログ出力、モニタリング可能
□ ドキュメント: README、API 仕様書、コメント
```

## 自動検証

```bash
# TypeScript
npm run type-check && npm run lint

# Go
go vet ./... && golangci-lint run

# Python
mypy src/ && pylint src/

# セキュリティ
npm audit / pip-audit / nancy sleuth
git secrets --scan
```

## レビュープロトコル

### セルフレビュー

```
コード: ロジック正確/エッジケース/エラーハンドリング/可読性/重複なし
テスト: 正常系/異常系/境界値、全 Pass
セキュリティ: 入力検証/SQLi/XSS/認証認可
パフォーマンス: N+1 なし/不要ループなし/メモリリークなし
```

### ペアレビュー

```
Developer Agent → QA Agent → 問題あれば差し戻し → なければ承認
```

## アンチパターン

| パターン | 重大度 | 対処 |
|---------|--------|------|
| `: any` 型 | 高 | 具体的な型 |
| `== null`（==） | 中 | `===` |
| 未サニタイズHTML挿入 | 高 | textContent / サニタイザー |
| SQLに変数直接展開 | 高 | プレースホルダー |
| God Object（>500行） | 中 | クラス分割 |
| Deep Nesting（>4） | 中 | Early return / Guard Clause |
| Magic Number | 低 | 定数化 |

## 品質基準

| 指標 | 基準 |
|------|------|
| 型安全性 | 100% |
| カバレッジ | ≥80% |
| 循環複雑度 | ≤10 |
| コード重複 | ≤5% |
| バグ・脆弱性 | 0 |
| 応答時間 | ≤200ms |
| セキュリティスコア | ≥90 |

## 事後分析（Post-Mortem）

```markdown
### 根本原因（5 Whys）
1. なぜ発生? 2. なぜその状態? 3. なぜ防げなかった?
4. なぜチェック見逃し? 5. なぜプロセス不十分?

### 再発防止
- 即座の対策 / 短期的 / 長期的
```

## 最終チェックリスト

```
□ セルフレビュー完了
□ 全テスト Pass / カバレッジ ≥80%
□ 型安全 100% / セキュリティ完了
□ パフォーマンス基準達成 / ドキュメント更新
□ エラー・アンチパターン未検出 / 品質ゲート Pass
```

全Passまで完了としない。**多層防御、継続的検証、失敗からの学習で高品質を保証。**
