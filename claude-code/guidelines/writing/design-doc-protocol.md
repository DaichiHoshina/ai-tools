# DesignDoc作成プロトコル

DD = DesignDoc。共通文章原則は [PRINCIPLES.md](PRINCIPLES.md) 参照。

## 前提: DDと詳細設計書は別物

混同するとレビュー不能になる。**書く前にWhyかHowかを判別する**。

| 項目 | Design Doc | 詳細設計書 |
|------|-----------|-----------|
| 主目的 | 設計判断の共有 | 実装仕様の定義 |
| 主読者 | エンジニア全体 / レビュアー / SRE / PdM | 実装者 |
| フォーカス | **Why / Architecture** | **How / Implementation** |
| 内容 | 背景・課題・選定理由・トレードオフ・非機能要件 | テーブル定義・API・SQL・バリデーション・シーケンス |
| 例 | 「なぜSKIP LOCKEDを使うか」 | 「SQLをどう書くか」 |

**DDD / マイクロサービス領域ではDD力 > 詳細設計力**。transaction / lock / idempotency / saga等は「実装」より「設計判断」が価値。

## DDで必ず書く6要素

| # | 要素 | 内容 |
|---|---|---|
| 1 | **Why** | PRD / 要件との接続 (×「新テーブル作る」○「O(1) 抽選のため新テーブル作る」) |
| 2 | **比較とトレードオフ** | Alternatives必須、最低2案。設計は正解でなく選択 |
| 3 | **変更耐性** | 「今動く」でなく「変更しやすいか」 |
| 4 | **責務境界** | service / module間の役割 (DDD的、曖昧だと事故る) |
| 5 | **失敗ケース** | 在庫不足・API失敗・二重実行・冪等性など本番で死ぬ点 |
| 6 | **移行戦略** | DB変更はExpand → Migrate → Contractの3段 |

レベル高い書き方: 数字で語る (O(n)→O(1)) / 図で説明 (シーケンス・ER・アーキ) / 制約明示 (MySQL 8.4 LTS、READ COMMITTED)。

## DDと runbook / 実装 docsの役割分担

DDは **方針** を扱い、**具体閾値・実装値・運用手順** は別 doc (runbook / 監視設定 / 実装コメント) に委譲する。

| 種別 | 扱う内容 | 例 |
|---|---|---|
| DD | 段階分け / 監視意図 / 設計判断 / トレードオフ | 「監視は 3 段階 (当日 / 観察 / 運用)」「なぜこの metricを取るか」 |
| runbook / 監視 doc | 具体閾値 / 通知先 / 対応手順 | 「CPU 80% で alert、Slack 通知先は opsチャンネル」 |
| 実装コメント | 時系列の制約 / API 仕様 / 値の根拠 | 「timeout=2sは p99 + 余裕」 |

**重複検知**: 同じ事象がDDと runbookで具体値含め説明されていたら、DD 側を方針記述に縮め、runbookへリンクで委譲する。DD 内で「具体値は <runbook 名> 参照」と書く。

**Why**: DDで具体値を書くと、運用調整で値変更時に DDも追従しないと食い違う。方針 (Why) と値 (How) を分離すると追従コストが値側だけに収まる。

## 4 Step

| Step | 概要 | 具体 |
|---|---|---|
| 1. 着手前リサーチ | 既存資産・前提の事前確認 | 5観点 + PRD見出し対応表 |
| 2. 初版必須要素 | テンプレ通りに埋める | テンプレ / 用語定義 / アクターmatrix / 数値4点 / Rollback / 未決 / 監視1:1 / UTC-JST / 環境固有名詞 |
| 3. 完成前3段スキャン | 提出前セルフレビュー | 抽象bullet / 数値根拠 / 用語 → lint実行 |
| 4. レビュー対応 | コメント対応規約 | 1コメ = 1 commit、返信にcommit URL、textlint単独commit禁止 |

### Step 1詳細: 着手前5観点

| 観点 | 確認内容 |
|---|---|
| 既存ダッシュボード | 監視・SLOがすでにあるか |
| フラグ | feature flag / kill switchの有無 |
| 識別子実在 | 想定するID・キー・enumが実在するか |
| DB前提 | テーブル・index・制約・サイズ感 |
| 関連層影響 | API / Worker / Batch / 別serviceへの波及 |

### Step 2詳細: 数値4点セット

「規模・頻度・遅延・コスト」を最低でも記載。記載不可なら「未測定 / 推定 / 仮定値」と明示。

### Step 3詳細: 3段スキャン

1. **抽象bullet**: 「対応する」「整理する」等の動詞だけのbulletを発見 → 具体化
2. **数値**: 「多い」「速い」「遅い」を見つけ → 数値or範囲に
3. **用語**: 初出の専門用語に定義があるか / コードフェンス羅列がないか
4. **評価の独立性**: 「複数チェックがある」=「安全」と判定する前に、各チェックが独立した source / 観点を参照しているか確認する。同一値を 2 経路で参照しているだけなら1チェック扱い

## テンプレ選択

| パターン | 状況 | 行数目安 |
|---|---|---|
| **軽量・単ファイル** | 小〜中機能、AI生成前提、レビュー負荷下げたい | ≤ 300 |
| **フル・単ファイル** | 大規模変更、セキュリティ影響、複数ドメイン跨ぐ | ≤ 800 |
| **分割型ディレクトリ** | 新サービス立ち上げ・大規模プロジェクト | 各 ≤ 400 |

**迷ったら軽量版から開始** → 行数超過や設計判断の複雑化でフル / 分割へ昇格。

### 共通ヘッダー

```markdown
# Design Doc: {タイトル}

Status: {draft | proposed | approved | archived}
Created: YYYY-MM-DD
Issue: {Issue URL}    # あれば
PRD: {PRD URL}        # あれば
```

### 軽量版テンプレ (5節)

```text
## 1. Overview        # 何を実現するか / Issue・PRD リンク
## 2. Data Flow       # シーケンス図 / 主要フロー
## 3. Data Schema     # テーブル設計 5 列表
## 4. API Design      # エンドポイント表
## 5. Appendix        # 代替案 / 未決事項 / 関連リンク
```

### フル版テンプレ (12セクション)

Overview / Goals & Non-Goals / Background / High-Level Design / Detailed Design / Alternatives / Trade-offs / Failure Handling / Migration Plan / Rollback / Observability / Open Questions

### 分割型構成

```text
DesignDocs/{context}/{project}/
├── summary.md       # エントリ、全体像・リンク集
├── architecture.md  ├── domain-model.md  ├── table.md
├── api.md           ├── sequence.md      ├── ui-*.md / batch.md
```

`summary.md` を最初に書く。分割基準は「担当エンジニア」or「レビュー粒度」が別れる単位。

## アンチパターン (ドラフト→最終稿で潰す)

| パターン | 対処 |
|---|---|
| 丁寧な説明口調 / 敬体常体混在 | 常体統一、結論→理由 |
| 箇条書き3階層以上のネスト | 文章化or 3段まで |
| 同内容反復 / 網羅列挙 | 1箇所集約、重要なものだけ |
| 自己参照 (「本DD」「実装PRで決める」) | 地の文 / 「実装時に決める」 |
| 実装詳細貼付 (関数順序・ファイルパス・SQL全文) | 「何をしたいか」まで、テーブル名+5列表+index bullet |
| 過去経緯・ドラフト造語 (「当初」「案A」) | 削除、比較は代替案表 |
| WHY欠落の制約 (「分割禁止」だけ) | 「守らないと何が壊れるか」1行 |
| 指示語参照曖昧 / 旧概念残存 / DB方言混同 | テーブル名・列名で一意化、新方式棚卸し、採用DB構文 |
| 既出議論との不整合 | 朝会・チャットの方針を反映、既出案は代替案表 |
| 実装状況マーカー (✅🚧❌) / 番号付き見出し | Statusヘッダー / `##` `###` で十分 |
| 削除残骸文・テンプレラベル残骸 | 最終読み直しで除去 |

## 3 段レビュー (内容 → 文章 → 読みやすさ)

DDレビューは観点が独立した 3 段に分け、**別ループで順に回す**。1 回で全部見ようとすると観点が混ざり、修正後に別観点でやり直しになる。

| 段 | 観点 | チェック対象 | 使うツール |
|---|---|---|---|
| 1 | 内容整合性 (docs) | ファイルパス / 構造体フィールド / DB NOT NULLカラム / PRD 受け入れ条件の対応抜け / Phase 計画の依存 / Why-not 表の妥当性 | `comprehensive-review --focus=docs` / 既存コードgrep |
| 2 | 文章規定 (writing) | 評価語 (重要 / 必須) の根拠併記 / 助詞重複 / だ・である調混入 / AI listing pattern / 抽象語 / 読後アクション末尾明示 | NG 辞書 + リポ textlintをローカル実行 |
| 3 | 読みやすさ (readability) | 情報密度 / 表 vs 段落バランス / 優先度の有無 / 図解の統一 / forward referenceのアンカー化 / 新規読者がスタックする箇所 | AI 目視 + 「初見の PM / 新規engineer 視点」シミュレーション |

**順序が固定の理由**: 内容 → 文 → 構造の順で抽象度が変わる。段 1で内容を直しても段 2で書き直しが入り、段 2で書き直しても段 3で構造変更が入ると、各段の観点が無効化される。

**運用**:
- `/design-doc` 出力後・PR 起票前に段 1
- PR 起票後・CI 通過後に段 2 (textlint / markdownlintの annotation 確認も含む)
- 最後に段 3 で「初見の読者が読めるか」を確認する
- 各段で修正したらコミット + pushを段ごとに分ける (commitメッセージで段がわかる)

## 読みやすさ改善パターン (段 3で使う)

DDを「初見の PM / 新規engineer / 運営」が読める形にする。1 件直すと連鎖的に他の違和感も消えることが多い。

| パターン | 適用条件 | 効果 |
|---|---|---|
| 用語表を概要より前に置く | 概要内に未定義の専門用語が登場 | 流し読みで意味が取れる |
| 図解の表記を統一する (Mermaidに揃える) | ASCII artと Mermaid 混在 | 視覚的に分断されない |
| 表のセル長文を要約 1 行に圧縮、詳細を表外段落へ | 表で scanできない | 一覧性と詳細性を両立 |
| 曖昧表現に注釈を追加 (「あり / なし両対応」等) | 専門用語回避で意味が消えた表現 | 「(値が入っていても、空 = NULLでも保存できる)」等 |
| 「最重要の制約」を独立 sectionとして冒頭に | 設計判断の多くの根拠が同じ制約に紐づく | 後の表の各セルから参照可 |
| 未決事項に優先度列 + 担当候補列 | 未決事項 5 件以上 | 読み手の次アクションが拡散しない |
| リスク一覧表 (重要度 + 対策難易度) を本文の前に | リスク 3 件以上 | 重要度の差を見せられる |
| リリース計画に依存列 + タイムライン図 | ステップ 4つ以上 | ステップ間の混在期挙動を可視化 |
| forward referenceを全部アンカーリンク化 | 「○○参照」が複数 | 遷移コストゼロ |
| シーケンス図のラベル短縮 | mermaidテキストが視覚的に詰まる | 図の可読性向上 |

**Why**: DDは新規読者が「approveするか / 自分の領域を確認するか」を判断する道具。情報を平等に並べると、優先度判断 + 次アクションが拡散する。表 / 図 / アンカー / 優先度の 4 武器で情報階層を作る。

**運用**: 段 3で AIが「初見の PM / 新規engineer / 運営」視点でスタックポイントを 10 個ほど洗い出す → 優先度 高 (構造変更) / 中 (図解差し替え) / 低 (リンク化) に分け、高から直す。

## 反復圧縮パターン (フィードバック別)

DDを「みやすく / 端的に / 抽象度を上げて」と反復で詰めるときに繰り返し効いたパターン。

| パターン | トリガー | 修正方向 |
|---|---|---|
| 役割ラベル付きbullet | 「わかりづらい」「インデント深い」 | 各 bullet 頭に「確認: / 追加待機: / 上限: / 確定:」等の役割語を付与し構造を平らにする |
| 表 → bullet 化 | 「表だとみづらい」 | **デフォルトは bullet**。表は「5 行以上 かつ 2 軸以上で機械的に対応する」場合のみ採用 |
| bullet → 表化 | 「散文で軸が見えない」 | 状況軸が複数あるなら表で「状況 × 手順」を示す |
| Whyを 1 行添える | 「これはなぜ?」 | 数値や運用ルールの後に **理由 1 行** を sub-bulletで追加。長文化禁止 |
| 単語だけの説明を撤去 | 「単語だけの説明やめて」 | 「比較基準を取得する」→「他シナリオを評価するための比較基準を取得する」等、目的語と動詞を明示 |
| 詳細値を別場所に逃がす | 「書きすぎでは?」 | しきい値・タイムアウト値等の具体値は未決事項表 / 別 repoに集約、本文は方針のみ |
| 重複・自明削除 | 全体読み直し | 「runtimeで旧方式に戻す手段は持たない」等の自明・前置きを削除 |
| 軸の統一 | 「時間軸 / 状態軸 / 復旧手段軸が混在」 | 軸を 1つに決めて分割、section 名を軸に揃える |
| 「絶対」確認 | 「絶対に実現できる?」 | 担保条件を明示。「主条件 + 補助条件 + 上限保証」のように multi-layerで書く |
| 抜けの再点検 | 「本当にこれでいい?」 | 直前で書いた選択肢が新セクションで活用されてるか検算 |

**反復順 (フィードバック別)**:

1. 初稿: 思いついた構造で書く
2. 「みやすく」: 表 ↔ bulletの入れ替え + 役割ラベル付与
3. 「端的に」: 詳細値を別場所に逃がす + 自明削除
4. 「これはなぜ?」: Whyを 1 行sub-bulletで追加
5. 「絶対?」: 担保条件を multi-layerで明記
6. 「本当に?」: 直前で書いた前提・選択肢の活用抜けを点検

**アンチパターン**:

- 表に集約しようとしすぎる → 行数少 / 内容短 は bulletの方が読める。**デフォルトbullet、表は最終手段**
- 「なぜ?」に長文で答える → 1 行sub-bulletを超えたら本文に書きすぎ
- 全 bulletに役割ラベル付与 → 装飾過剰。インデント深い箇所だけ
- フィードバック直後の全体書き直し → 該当箇所だけ局所修正、波及は確認してから

## 図表の型を固定する

| タイプ | 形式 |
|--------|------|
| 処理フロー | mermaid sequenceDiagram (User / API / DBの3-4 actor) |
| アーキ図 | mermaid flowchart (ASCII罫線はNG) |
| テーブル設計 | カラム名 / 型 / NULL / デフォルト / 説明 の5列 |
| インデックス | bullet (主キー → 複合 → UNIQUEの順、各に「なぜ」1行) |
| エンドポイント | メソッド / パス / 説明 (必要なら「変更内容」列) |
| 代替案 | 方針 / 採否 / 理由 の3列、採用 ○ / 却下 × |
| リリース手順 | 番号付きリスト (コマンド / 操作者 / 戻し方) |
| 未決事項 | # / 決定タイミング / 項目 / 決定方法 の4列 |

### mermaidシーケンス図の micro 規約

- **alt 複数は点線が見づらい**、1 個に統合し失敗分岐は alt 内で完結させる
- actor 数は 3-4に絞る (User / API / DB / 別 service 程度)
- ラベルは短文化、図内に英文長文を入れない (詳細は本文に書く)

## 想定SQLの書き方

コード全文は貼らない。意図のみbulletで。

| 項目 | 書く内容 |
|---|---|
| 対象 | テーブル・範囲 |
| 条件 | WHERE節の論理条件 |
| 並び順 | ORDER BYの意味 |
| 件数 | LIMIT |
| ロック | FOR UPDATE / SKIP LOCKED等の挙動 |

## 数値の書き方

- **OK**: ローカルベンチマーク、本番環境スペック (IaC参照)、本番データ集計 (公開API・メトリクス)
- **NG**: 仮定スループットから逆算した近似値、未検証の仮定値

仮定しか書けないものは未決事項に逃がす。

## 専門用語の置換 (読者層に応じて)

| 専門用語 | 置換案 |
|---|---|
| Fisher-Yates | ランダムな順 |
| Parallel Change | 段階移行 |
| O(1) | 件数に依存しない |
| 冪等性ガード | 二重実行防止 |
| 同期ブロッキング | 完了まで応答を返さない |
| スループット | 書き込み速度 |

残す: `p95` / SQL構文 (`SKIP LOCKED` / `FOR UPDATE` 等) / クラウド・OSS固有名詞。

### section 別に用語層を切替える

1つの DD 内でもsectionごとに読者層が異なる。サマリ系 sectionは PM / 他チーム engineer 想定で日本語優先、技術詳細 sectionは同領域 engineer 想定で英語慣用を許容する。

| section 性質 | 読者層 | 用語方針 |
|---|---|---|
| Overview / 監視 / 検証結果 / 結論 | PM / 他チーム engineer / SRE / QA | 日本語優先 (専門用語は初出で平易化) |
| 詳細設計 / 並行制御 / DB 内部 / SQL | 同領域 engineer / レビュアー | 英語慣用 OK (locking read / MVCC snapshot 等) |

**判断**: section 冒頭で「この sectionは誰が読むか」を 1 秒で判定して用語層を選ぶ。同section 内での揺れは禁止 (同じ単語が日本語と英語で混在する状態は最悪)。

### DB / 並行制御 領域の置換例 (架空)

| 専門用語 | 平易化置換 |
|---|---|
| MVCC snapshot | 単独の SELECT 文 |
| locking read | UPDATE 内 SELECT |
| 暗黙の Sロック | 自動取得される共有ロック |
| Xロック / Sロック | 排他ロック / 共有ロック |
| stale-write race | 古い値で書き戻す競合 |
| innodb_lock_wait_timeout | MySQLのロック取得待ち |
| Error 1213 | MySQLのデッドロックエラー (1213) |
| index range scan | 複合インデックスで効率的に検索 |

**残す**: SQL 識別子 (`FOR UPDATE SKIP LOCKED` / `READ COMMITTED`) / クラウドOSS 固有名詞 / 数式 (`O(1)`)。共有sectionでは平易化、技術詳細 sectionでは英語残置 (「section 別に用語層を切替える」参照)。

## 未決事項の書き方

「項目 + 決定方法」を対にして書く。決定方法がTBDなら表に載せず削除。

| 本文に書く | 未決に逃がす |
|---|---|
| 合意済みの方針 | 合意前の提案 |
| コード・計測で検証済みの事実 | 仮定・推定 |
| 今回やること | 別issueでやること |

## レビュー対応 (Step 4詳細)

レビュー応答規約 (1コメント = 1 commit / 返信に commit URL / Resolve 主体) は `pr-description.md` を canonicalとする。DD 固有の追加観点のみ以下に記載。

- **3 分類対応**: 設計判断変更 = 本文書き直し / 説明不足 = 段落補足 / YAGNI 懸念 = 削除 or 未決事項へ
- 派生した未合意論点は未決事項にリンク

## セルフチェック18項目 (Step 3完成前)

```
[ ] スコープは「画面・機能」単位
[ ] 全主張に「誰が決めたか / どう検証したか」の根拠
[ ] 数値に実測・計測・スペック参照の根拠
[ ] 代替案の不採用理由が 1 行で説明できる
[ ] 未決事項は「項目 + 決定方法」で対
[ ] bullet 語尾統一、敬体・常体混在なし
[ ] 「PR」「本 DesignDoc」等の自己参照なし
[ ] 「〜は存在しない」のゼロ事実なし
[ ] 関数順序・具体ファイルパス等の実装詳細なし
[ ] 表記・節構成が同カテゴリ既存 DD と揃う
[ ] ドラフト中の造語・過去経緯文なし
[ ] 編集中の孤立文なし
[ ] 制約・禁止事項に「なぜ」1 行
[ ] 「将来〜の場合」に発生条件併記
[ ] 指示語の参照先が一意特定可
[ ] 旧概念 (リトライ上限・フォールバック等) の残存なし
[ ] SQL が採用 DB の方言
[ ] 既出議論と本文方針が整合
```
