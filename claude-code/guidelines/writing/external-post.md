# 外部発信文章 (PR コメント / Slack / Issue / Notion 共通)

[pr-description.md](pr-description.md) の原則を「PR コメント / レビュー依頼 / Slack 返信 / Issue コメント / Notion 投稿」にも適用する。

## 起点

- **完成形先出し禁止**: 草稿を一気に出さず、**観点・方向性を先にユーザーに提示して合意**してから本文化する
  - Why: 完成形コメントを先出しすると、観点取捨選択の手戻りが発生する
  - 例外: 緊急アラート対応で時間制約があるとき (それでも事後に観点を残す)

## 構造

- **TL;DR + 詳細の 2 階層** (重点 / 参考、must / fyi 等)
- 各論点は「**現状 → 取った対応 → 確認したい点**」のストーリー型に統一
- 散文段落を排し、箇条書き + インデントで関係を視覚化

## 用語

- **コード識別子をコードフェンスで並べない**、自然文に溶かす
- 専門用語は最小限、レビュアーが詳細を知らない前提で書く
- 技術語は平易語に置換

| NG (用語羅列) | OK (自然文化) |
|---|---|
| `clone_products.sql` の `ORDER BY id` を `migrate_slots` の `rand.Shuffle` に揃える | 並び順を本番に揃える方式に統一した |
| `BASE_URL` / `BRANCH_NAME` 未指定で init context throw | 接続先と環境タグを未指定で起動した場合に停止する |
| VU / SKIP LOCKED / DB ロック | 同時実行ユーザ / 先頭から空き行を順に取る仕組み / DB のロック競合 |

## 文長・記号

- **1 文 60 字以内**、段落 3-5 文を上限の目安
- 過剰な記号 (🙏 / — の連発 / 顔文字) は排除
- 太字見出し + 箇条書きで scan しやすく

## 最終チェック (投稿前)

5 軸で自己評価。プロジェクトに lint 環境があれば併用:

| 軸 | 観点 |
|---|---|
| 可読性 | 1 文の長さ・段落の塊感 |
| 視認性 | 階層・記号・太字の使い方 |
| 伝達性 | 知識前提の最小化、専門用語の平易化 |
| 根拠性 | 数値・URL・commit リンクの有無 |
| 整合性 | section 重複なし、起点の合意通り |

15 点満点で **11 以上** を合格ライン (プロジェクト基準があればそちら優先)。

## 投稿前ワークフロー

1. 観点・方向性をユーザーに提示
2. 合意後、本文ドラフト
3. 5 軸セルフレビュー
4. ユーザー最終確認
5. 投稿 (返信先 / 宛先誤りに注意)
