# 外部発信文章 (PRコメント / Slack / Issue / Notion共通)

共通文章原則は [PRINCIPLES.md](PRINCIPLES.md) 参照。

[pr-description.md](pr-description.md) の原則を「PRコメント / レビュー依頼 / Slack返信 / Issueコメント / Notion投稿」にも適用する。

## 起点

- **完成形先出し禁止**: 草稿を一気に出さず、**観点・方向性を先にユーザーに提示して合意**してから本文化する
  - Why: 完成形コメントを先出しすると、観点取捨選択の手戻りが発生する
  - 例外: 緊急アラート対応で時間制約があるとき (それでも事後に観点を残す)

## 主管領域チームへの相談 (SRE / DA / Security / Infra など)

主管領域を持つ他 team に相談する Slack / Notion / issue の文章は「断定的依頼」より「認識共有 + 開いた相談」で書く。**「目的 + 依頼」形式にすると相手の判断余地を奪い、押し付けに見える**。

### 構成

1. **背景**: 状況説明 + 認識共有 (「〜と考えています」「〜と認識しています」)
   - センシティブ性 / リスク / 制約などの自分側の理解を理由付きで開示する
2. **現状認識**: 前提条件の箇条書き + 「相違あれば指摘いただきたいです」
   - 自分の理解が誤っている可能性を明示し、訂正余地を残す
3. **ご相談**: 課題提起 1 文 + 「どう対応するのが良いでしょうか」+ 具体的な確認点 3-4 点
   - 「依頼」「目的」と書かず「ご相談」「教えていただけると助かります」

### トーン

| 避ける | 使う |
|---|---|
| 目的: 〜すること | 〜と考えています / 認識しています |
| 依頼: | ご相談: |
| 〜してください | 〜していただけると助かります |
| 〜の手順を提供 | 〜の手順を教えてください |
| (前提を断定) | (前提) と認識しています、相違あれば指摘いただきたいです |
| 必須 / 必要 / すべき | 必要があると認識しています / 〜と考えています |

## 構造 (ストーリー型)

短文向けの基本構造 (PREP 3点や媒体別構造の詳細は [PRINCIPLES.md](PRINCIPLES.md#媒体別構造) 参照):

- **TL;DR + 詳細の2階層** (重点 / 参考、must / fyi等)
- 各論点は「**現状 → 取った対応 → 確認したい点**」のストーリー型に統一
- 散文段落を排し、箇条書き + インデントで関係を視覚化

## 用語の自然文化

- **コード識別子をコードフェンスで並べない**、自然文に溶かす
- 専門用語は最小限、レビュアーが詳細を知らない前提で書く
- 技術語は平易語に置換 (詳細置換表は `references/writing-patterns.md` 参照可、簡易は下表)

| NG (用語羅列) | OK (自然文化) |
|---|---|
| `clone_products.sql` の `ORDER BY id` を `migrate_slots` の `rand.Shuffle` に揃える | 並び順を本番に揃える方式に統一した |
| `BASE_URL` / `BRANCH_NAME` 未指定でinit context throw | 接続先と環境タグを未指定で起動した場合に停止する |
| VU / SKIP LOCKED / DBロック | 同時実行ユーザ / 先頭から空き行を順に取る仕組み / DBのロック競合 |

## GitHub auto-link 衝突を避ける

PRコメント / Issue / Slackで `#1234` 形式の番号を本文中に書くと、GitHubが同repoの issue / PRへの auto-linkを貼る。意図しない link 接続を防ぐため:

| 場面 | 書き方 |
|---|---|
| 同repoの issue / PRを参照 | `#1234` (auto-link 期待) |
| 他 repoの issue / PR | `owner/repo#1234` (cross-repo auto-link、repo 名明示) |
| DD 内部の section 参照 (例: §9-2 #5) | `` `#5` `` (バッククォートescape、auto-link 抑止) |
| 番号だけの参照 (例: 「項 #3」) | `` `#3` `` (escape) |

**判断**: 番号 literal を書く前に「当該番号は GitHub issue / PR の番号か、本文内の項番号か」を判定し、項番号なら必ず escape する。

## Issue 起票時の追加規約

bug report / feature request の Issue は、コメントより構造化が必要。下記 3 点を必ず満たす。

| 項目 | ルール |
|---|---|
| **再現手順** | 3 ステップ以内に圧縮する。4 ステップ以上は前提条件が混在しているサイン、別 Issue 分割を検討 |
| **期待値 / 実測値分離** | `## 期待` `## 実測` を必ず別 section に分ける。混在すると判断と事実が混じり、再現確認が止まる |
| **環境情報** | OS / app version / commit hash / 関連 config を冒頭 1 ブロックに固める。bug 系は特に必須 |

feature request は `Why (現状の不便) / What (期待動作) / How (案がある場合のみ)` の 3 セクション。How が無い request も受け入れる (議論余地を残す方が良い案が出る)。

## 文長・記号

- **1文60字以内**、段落3-5文を上限の目安
- 過剰な記号 (🙏 / — の連発 / 顔文字) は排除
- 太字見出し + 箇条書きでscanしやすく

## 最終チェック (投稿前、5軸採点)

| 軸 | 観点 |
|---|---|
| 可読性 | 1文の長さ・段落の塊感 |
| 視認性 | 階層・記号・太字の使い方 |
| 伝達性 | 知識前提の最小化、専門用語の平易化 |
| 根拠性 | 数値・URL・commitリンクの有無 |
| 整合性 | section重複なし、起点の合意通り |

**採点**: 各軸1-3点 (1=不足 / 2=可 / 3=良) で15点満点。**11以上を合格ライン** (プロジェクト基準があればそちら優先)。簡易運用なら全5軸2点以上 (= 各軸pass) の二値判定でも可。

共通セルフチェックは [PRINCIPLES.md](PRINCIPLES.md#出力前セルフチェック-7項目6項目以上で合格) 参照。Web 出力時は `## Web 可読性` の +4 項目も適用。

## 投稿前ワークフロー

1. 観点・方向性をユーザーに提示
2. 合意後、本文ドラフト
3. 5軸セルフレビュー (+ 共通セルフチェック7)
4. ユーザー最終確認
5. 投稿 (返信先 / 宛先誤りに注意)

## Issue コメントで調査結果を共有する書き方

技術調査結果 (Datadog / APM / 負荷試験結果など) を Issue コメントで PM / 非エンジニアも読む状態で共有するときは、**「実測値 → つまり (解釈) → アクション」の 3 段構成**にする。技術データだけ並べると読み手が判断できず意思決定が拡散する。

### テンプレ

```markdown
## XXX 分析 (YYYY-MM-DD 追加調査)

[調査のきっかけと範囲を 1-2 行]

### 実測値

| 項目 | 実測値 | 評価 |
|---|---|---|
| ... | ... | ⚠️ 異常 / ✅ 余裕 |

### つまり

[技術用語抜きで、何が起きていたかを平易な日本語で 2-3 段落]
[可能なら前向きにフレーミング (「最悪ケースでも壊れない強い保証」等)]

### アクション

| 確認項目 | 内容 | 優先度 |
|---|---|---|
| ... | ... | P1 緊急 / リリース前確認 |

[最後に 1 文で締める]
```

### 効くパターン

- **「つまり」段落で意訳**: 「試験はアプリサーバー 1 台だけで動いていた状態」のような平易表現
- **異常を前向きにフレーミング**: auto-scale 不発 (異常) → 「最悪ケースでも壊れない強い保証」
- **平易な比喩**: 「片肺運転」「詰まる」「1 台だけで動いていた」
- **表で raw data、地の文で解釈**: 数値は表に閉じ込め、本文は判断・意思決定に集中
- **評価列の絵文字**: ⚠️ 異常 / ✅ 余裕 で視覚的に判定を示す
- **前向きな結語**: 「これらが想定どおりであれば〜」で締める。恐怖を煽らず行動を促す

### やってはいけない

- 技術データだけ表で並べて解釈なしで終わる (読み手が判断できない)
- 異常発見を負の表現だけで書く (「壊れる可能性」だけだと意思決定が止まる)
- 1 セクションに raw data + 解釈 + アクション全部詰める (3 段に分ける)
