# 文章執筆の共通原則

ヒト向けドキュメント共通原則。個別ルールは各 file ([commit-message](commit-message.md) / [pr-description](pr-description.md) / [external-post](external-post.md) / [long-form-doc](long-form-doc.md) / [design-doc-protocol](design-doc-protocol.md)) 参照。

## 最優先 — 読み手の認知負荷を下げる

ドキュメントは記憶ダンプでなく、判断 / 行動を助ける道具。すべてのルールはこの目的から派生する。

- 1回読めば理解できる文章を目指す
- 読み手に「で、つまり何？」と思わせない
- 「賢そう」でなく「伝わる」を優先する
- 初見の人が途中で止まらないこと

迷ったらこの4つに戻る。判断の優先順は「ルール準拠」より「読み手の負荷」。

### 専門語・翻訳調・カタい語を使わない (読み手知らない語禁止)

書き手が知ってる語 ≠ 読み手が知ってる語。中学生でも分かる日常和語を選ぶ。迷ったら短い和語。

- × 外挿 / 余裕大 / 無視可 / 無視可能 / 懸念解消 / 全観点 / 判定確定 / 漸近 / 線形 / 冪等性 / 直交 / ロバスト
- ○ 本番だと / 枠の半分以下 / 気にしなくていい / 心配なし / 全部OK / GOで進める

技術用語 (TX / migration / replica lag / index 等) は読み手 (engineer / PM) の前提知識として残す。**書き手の語彙でなく読み手の語彙で書く**。

書く前の self-check: 「この語、user が知らないと言ったら言い換えられるか？」答え No なら使わない。

## 書く前の4問

抽象的な答え不可。答えずに書くとAI文体になる。

1. **読み手は具体的に誰か** (×「エンジニア」○「新joinのsenior backend」「レビュアーAyanoさん」)
2. **読後に何を決めさせる / やらせるか** (×「理解させる」○「approveを決めさせる」「migrationを実行させる」)
3. **主張を裏付ける数字or事例** (×「大幅改善」○「p99 1.2s → 320ms (-73%)」)
4. **「なぜ？」と聞かれたら** (×「〜が必須」○「localStorageにtoken置かない、XSSで抜かれる」)

## 守る指針 (7原則)

1. **用語を並べるだけで説明しない**: 専門用語の羅列は内容ゼロ。1用語ごとに「何のために・どう動くか」を1文添える
2. **暗黙知を前提にしない**: 「いつもの」「ご存知の通り」を使ったら、その内容を1文で書く
3. **次の疑問を先回り**: 1段落書いたら「読み手が次に何を聞きたいか」を想像し、次段落に答えを置く
4. **見出しだけで流れが分かる**: 見出しはラベル (「アーキテクチャ」) でなく主張 (「読み書きを分離して負荷を分散する」)
5. **段落に役割を持たせる**: 各段落は「背景 / 理由 / 具体例 / 結論 / 注意点」のどれかに該当。役割が曖昧なら削除or分割
6. **抽象説明の直後に具体例**: 抽象名詞や原則を書いたら、次の1-2文で具体例 (数値・事例・コード断片)
7. **「賢そう」より「伝わる」**: 平易で短い文を選ぶ。読み手は実務判断のために読む

## 状態変化に追従して section 名を更新する

docs section は **現状を表す名前** にする。状態が変わったら section 名 / 表 column 名も更新する。

例 (架空):
- 試験前: 「未決事項」表 (status: in_progress 列あり)
- 試験完了後: 「検証結果」に rename、status 列削除 (完了で進捗 enum 不要)
- 仕様確定後: 「仕様」に rename、判断経緯は別 section に逃がす

**判断**: section 名が現状と乖離した瞬間に rename + cross-ref 更新。「未決事項」のまま中身が確定値の表になっていると読者が混乱する。

rename に伴い他 file / 同 file 内 cross-ref も `[未決事項 #X]` → `[検証結果 #X]` のように全置換する。

## 媒体別構造

### 短文 (issue / PRコメント / Slack / Notionコメント / ticket)

**PREP 3点構造**:

1. 結論: 読み手に何を判断 / 実行させるか (1行)
2. 理由: 現象 / 影響 / 原因
3. 次アクション: 担当 / 期限 / 不明点

- 本文400字前後 (30秒で読める)、title 80字前後
- 詳細ログ・スタックトレースは `<details>` 折りたたみor別ファイルリンク
- 詳細は [external-post.md](external-post.md) 参照

### 長文 (Design Doc / PRD / RCA / Notionページ)

- **TL;DR冒頭1-3文で結論**。「本稿では〜について述べる」系導入は削除
- **報告型doc (負荷試験 / 計測 / 障害 / 検証 / KPI共有) は冒頭3行で「最大値 / 測定条件 / 次アクション」必須**。最大値=計測した上限/閾値の具体数値、測定条件=どの環境/負荷/期間で測ったか、次アクション=読み手が approve / scale / 修正 のどれを決めるか。数値を結論行の本体に置き、別セクションに散らさない。
- 800-1500字 / ページ、2000字超なら分割
- 1セクション = 地の文3-5割 + 箇条書き / 表5-7割
- 箇条書きは3-5項目。7超は表or段落分割
- コードブロックは5行以内
- 詳細は [long-form-doc.md](long-form-doc.md) 参照

### PR description

4セクション固定: **Why** / **What changed** (数字・事例) / **Testing** / **Review focus**。詳細は [pr-description.md](pr-description.md) 参照。

### DesignDoc

12セクションor軽量5節。詳細は [design-doc-protocol.md](design-doc-protocol.md) 参照。

## PREP 法 + 5W1H (長文構造の規範)

長文出力は **PREP 法** で構成し、**why / how** を必須とする。

### 構造

- **P**oint (結論): 一行で何が起きたか / 何を変えたか
- **R**eason (理由): なぜそれが必要だったか (incident / 不便 / 目標)
- **E**xample (具体例): how の詳細、数値、動作、file path
- **P**oint (再確認): 締めの一行、効果や次の手

各 P/R/E/P は **太字 label** で明示する。暗黙構造は読み手に伝わらない、隠さない。

### 5W1H チェック

| 観点 | 充足例 | 欠落時の症状 |
|---|---|---|
| What | 「`/review` を 2 段階に変更」 | 「強化した」だけで何かわからない |
| **Why** (必須) | 「subagent 任せで取りこぼし発生」 | 「読み手が判断不能」「で?」と返される |
| **How** (必須) | 「parent が 7 観点で再 check、全 mode 適用」 | 抽象論で再現性なし |
| When/Where | commit hash / 日付 / file path | 通常 hash で足りる |
| Who | (省略可、necessary 時のみ) | 通常 commit author で足りる |

### 禁止 / 推奨

- **抽象語禁止**: 軸 / 層 / 経路 / 土台 / 狙い / 方針 → 具体動作 / 数値 / file path に置換
- **1 段落 ≤ 4 行**: 超える場合は箇条書きに分解
- **接続文削減**: 「背景は〜」「〜の延長線上にある」など本題と関係ない補足は削る
- **文末統一**: 体言止め混在を避け、外向き文書は「〜する / 〜した」で統一
- **同一文末の3連続回避**: 「〜した。〜した。〜した。」のように同じ語尾が3文以上続くと単調になる。常体は保ちつつ「〜した / 〜できた / 〜している / 〜となった / 〜する方針をとった」など語尾を散らす

### 例 (commit まとめ)

```text
**結論**: `/review` を 2 段階 self-check に変えた。
**理由**: 1 段階だと subagent 任せで見落としや過剰指摘が残る。
**例**: parent Opus が 7 観点で再 check。全 mode 適用。file: `commands/review.md`
**結論再確認**: subagent 単独判定の取りこぼしを parent が拾える状態。
```

### decision 要求時の Point 変種 (decision-frame-first)

応答が user に decision を求める場合、Point は調査結果でなく **決定の枠** を提示する。trigger は (1) 末尾が `?` で締める質問 (2) 本文に選択肢 (A/B、案 1/2、Yes/No、どちらにする、どう進める 等) を含む — 両方満たす応答全て、長さ閾値なし (短文 3 行でも適用)。

冒頭 1 行に `要決定: <選択肢の枠> / <候補数>` を置く。

```text
要決定: 子 project memory が parent を継承するか (Yes/No → 子削除可否)
情報: W2 完了 (16 file 削除)、統合方針は 1 通り、最終確認 1 点

要決定: 2 widget × 各 2 案 = 4 通り (A/B 独立、A は 1 つ選ぶ / B は 1 つ選ぶ)
情報: 両 widget とも attribute 不在で 0 件確定
```

**Why**: 真 `どういうこと?` 3 件 (2026-05-28 / 05-30) の root cause は「冒頭結論が `調査結果` を語り、user が返すべき内容 (decision の枠) が末尾質問に隠れていた」。長文冒頭結論 rule は trigger 満たしても「結論=調査結果」を許す書き方で素通り。短文ケース (3 行) は長文 rule の trigger 外で素通り。詳細: `~/.claude/references-private/analysis-doukouiukoto-pair-20260603.html` (private 保管、social hit 含有のため public repo 外置き)

## chatとdocumentで文体を分ける

| 場面 | 方針 |
|------|------|
| Chat応答 (genshijinモード等) | 体言止め可、助詞省略可、結論先出し、短く速く |
| ヒト向けdocument | 文として完結 (〜する / 〜した)、主語明示、地の文と箇条書き混在、常体統一 |

**document基本ルール**:

- 主語必須 (「担当者は〜」「ユーザーは〜」)
- 指示語禁止 (「これ」「それ」「上記」「前述」→ 具体名)
- 略語禁止 (初出はフルスペル)
- 「適宜」「状況に応じて」禁止 → IF-THEN形式で明文化
- 日付YYYY-MM-DD、英数字半角

### chat 草稿 → 外向き翻訳の 8 観点 sweep

chat の体言止め圧縮をそのまま外向き (PR / Slack / Notion / 報告) に流用しない。
下書き完成後、以下 8 観点で sweep してから提示する。

1. **体言止め排除 (常体統一)**
   - NG: 「3.5-7 分で完了見込み」「余裕大」「全観点クリア」「追加リスクなし」
   - OK: 「3.5〜7 分で完了する見込みになる」「十分な余裕がある」「いずれの観点でも問題は発生しない」「追加のリスクはないと判断する」
2. **略記を具体数値・正式名称化** — 詳細は AI 臭 3 変換 (a) 参照
3. **難解漢語・jargon 排除** — 「外挿」「諸元」等 → 平易表現。あわせて **漢字 5 字以上の連続を作らない** (例: 「段階的強化」→「段階的に強化」、「逐次化検出」→「逐次に並ぶ状態を検出」、「絶対禁止徹底」→「禁止を徹底する」)。固有名詞・技術用語は対象外、漢語連結のみ対象。機械検出は `grep -oE '[一-龯]{5,}'`
4. **指示語・矢印・記号 → 文展開**
   - NG: 「上記」「これ」「それ」「→ 本番でも数秒〜10 秒」
   - OK: 「前項の設定値」「<具体名>」「〜のため、本番でも数秒から 10 秒程度で完了する」
5. **主語明示** — 「実走した」→「<実施者> が実走した」または「テストとして実走した」
6. **表のセル内も単位明示**
   - NG: `188K×1` / `9秒` / `mismatch=0`
   - OK: `188,000 行 × 1 件` / `9 秒` / `mismatch=0 件`
7. **曖昧名詞は範囲明示**
   - NG: 「テスト完了」「動作確認」
   - OK: 「inventory_batch と notification_service を本番想定手順で実行した」
8. **AI 定型語 / カタカナ造語の事前 sweep** — NG 辞書から grep

詳細運用は「[## 機械検出 grep (出力前 sweep)](#機械検出-grep-出力前-sweep)」section と連動する。

### UI コピー / エラーメッセージ向け追加規約

ユーザー向け UI 表示文言・エラーメッセージは「やさしい日本語」基準を適用する。

- **1 文 40 字以内** (技術文書 100 字 / web 60 字とは別の最厳格 layer)
- 難読漢語を避け、小学校 4 年生程度の語彙を選ぶ
- 専門用語の代わりに具体動作を書く (NG: `認証エラー` / OK: `ログインに失敗しました。パスワードを確認してください。`)
- 一文一意 (複文 / 並列禁止)

出典: 総務省「やさしい日本語ガイドライン」。在留外国人 / 高齢者 / 児童含むユーザーが対象の場合は本規約を最優先。

## 構造原則 (docs 観点)

- **散文より bullet ファースト**: 説明は bullet で書く、地の文は接続・背景・結論のみ
- 各セクション直下に **目的 1 行** を添える
- 強調太字は GO/NO-GO 基準等の **限定箇所のみ** (濫用で埋没)
- forward reference は **anchor link 化** (「○○参照」より `[○○](#xxx)`)
- 用語は **他参照で再定義しない** (用語ポインタ表に統一)
- 環境・テストデータ等は **別 SoT 参照**、本書では再記しない

## 箇条書き階層化

- 因果連鎖・所属関係・補足説明は **親子で表現** (平坦に並列禁止)、2 スペースインデント統一
- 同一階層に並列するのは **対等な並列のみ**

## ミクロ規則

- sentence-length **120 字超は分割**
- 助詞重複 (「のの」「をを」「がが」) 排除
- 絶対表現 (「必ず」「絶対に」「完全に」) は GO/NO-GO 基準時のみ、説明文では緩和
- セクション記号 `§` は使わない (日本語では「Phase 2」直書きで足りる)
- 「埋める」より「記録する」「測定値を記入する」 (docs 寄り)
- web 出力時は `## Web 可読性` の 60 字上書き適用

## Web 可読性 (scan 前提)

外向き prose は web (GitHub / GitLab / Notion / Slack / Confluence) で読まれる。読者は読まずに **scan する** (NNG: 79% scan / 16% line-by-line)。chat 応答と別の構造が要る。

### scan pattern 対応

| pattern | 読者行動 | 書き手の対応 |
|---|---|---|
| F-pattern | 上から左寄り、段落先頭数語のみ拾う | 段落先頭に keyword、左端に情報 |
| layer-cake | 見出しだけ拾って必要箇所だけ本文へ | 見出しを「主張」化、descriptive に |
| spotted | 太字 / link / 数値だけ拾う | keyword を太字、評価語に数値併記 |

外向き文書生成時 (`/git-push --pr` / `/post-comment` / `/docs`) は 3 pattern の **どれで読まれても主旨が伝わるか** を self-check する。

### Web 用 ミクロ規則

- **1 文 60 字以内** (最大 80 字)、読点 3 個まで — 「## ミクロ規則」120 字を web 文脈では上書き
- **1 段落 3-4 行 / 250 字以内** — 段落間に空行必須
- **漢字比率 3 割目安** — 「行う / 出来る / 事 / 物 / 為」→「やる / できる / こと / もの / ため」
- **見出しは主張化** — 「アーキテクチャ」 NG / 「読み書き分離で負荷分散」 OK
- **結論先出し + 本文量半減** (inverted pyramid)
- **数値 / file path は太字 or `code`** — spotted pattern で拾われる確率上昇
- **見出し / キャッチは意味単位で改行** — Notion / Slack は実 viewer で preview
- **コードブロック / list の前後に空行** — renderer 差で潰れる対策
- **heading 前 2 語に keyword (front-loading)** — F-pattern で末尾は読まれない。先頭語が情報の核になるよう語順反転 (NG `データ可視化による不正検出の早期化` / OK `不正を 24% 早期検出: データ可視化導入`)
- **heading の文脈外自己完結** — search result / SNS feed / RSS で heading のみ露出するため、具体的事実 (数値 / 対象) を含める
- **heading の読点 3 個以下** — heading は 1 メッセージに絞る、句読点最小化
- **a11y: 色 + 書体 / 文言の二重識別** — 強調はボールド + 色、必須項目は「赤字」でなく「赤い ※ のついた」、色弱者 / screen reader 利用者に届く
- **WCAG 3.1.5 reading level**: 中学卒業相当 (連続ひらがな + 漢字長の計算式) を超えるなら平易版 or 要約ページを併記
- **モバイル: 難解語は読速を落とす** — comprehension は維持されるが読速低下で離脱する。SP 想定なら語数削減 + 平易語選択

### 出力前 web 用 追加 check (既存 6 項目に追加)

- [ ] 1 文 60 字以内 / 読点 3 個以内
- [ ] 見出しが主張型 (ラベル型でない)
- [ ] 段落 3-4 行以内、空行で区切り
- [ ] keyword / 数値 / file path が太字 or `code` で scan 可能

## 逆に考えるミニマル化

「何を削るか」より「**essence で何を残すべきか**」から発想する。

- 別 SoT で代替できる箇所は積極的に削除し、リンク参照に切替
- 環境構築 / 撤去 / 用語定義は外部 SoT に寄せる
- 本文書固有の価値 (GO/NO-GO 閾値 / 結果記録欄 / 当日手順) のみ残す

## 文単位の品質規約

`PRINCIPLES.md` 既存規約は段落 / 媒体構造を扱うが、本 section は **1 文単位の品質** を定量規範化する。textlint preset (ja-technical-writing / JTF-style) の運用知見を取り込む。

### 文長 (媒体別 2 段)

- **技術文書本文 (DD / RCA / long-form-doc)**: 1 文 100 字以内
- **web / 短文 (PR / Slack / Notion)**: 1 文 60 字以内 (`## Web 可読性` で既出)

文単位の上限は媒体で切替。100 字超は分割の合図。

### 読点 3 個まで

1 文中の読点 (`、`) は 3 個まで。超えたら文分割。読点 4+ は読者の認知負荷が急上昇する。出典: textlint-rule-preset-ja-technical-writing (max-ten rule)。

### 連続漢字 4 文字上限

漢字 5 文字以上の連続は読解負荷が高い。固有名詞・技術用語 (e.g. `Opus`、`hook`) は例外、漢語連結のみ対象。

| Before (連続漢字) | After (助詞挿入 / 訓読み開く / 動詞化) |
|---|---|
| `利用者認証処理管理` | `利用者の認証処理を管理` |
| `構造的変化傾向分析` | `構造的な変化傾向の分析` |
| `段階的強化` | `段階的に強化` |
| `逐次化検出` | `逐次に並ぶ状態を検出` |
| `絶対禁止徹底` | `禁止を徹底する` |

機械検出: `grep -oE '[一-龯]{5,}'` で外向き prose を sweep する。

出典: textlint-rule-preset-ja-technical-writing (max-kanji-continuous-len rule) を 2026-06-08 に 6→4 へ厳格化。

### ひらく漢字 7 品詞 (ひらがな表記推奨)

外向き prose では下記 7 品詞をひらがな表記する。記者ハンドブック (共同通信社) 業界標準に準拠。具体物・物体・時刻そのもの等の本来の意味で使う場合は漢字維持、抽象・概念・形式語の場合のみひらく。

| 品詞 | NG (とじる) → OK (ひらく) |
|---|---|
| **形式名詞** | する事 → すること / する時 → するとき (場合の意) / する為 → するため / する上で → するうえで / の様 → のよう / の通り → のとおり / する所 → するところ |
| **副詞** | 既に → すでに / 例えば → たとえば / 是非 → ぜひ / 沢山 → たくさん / 全て → すべて / 様々 → さまざま / 暫く → しばらく / 大分 → だいぶ / 恐らく → おそらく / 色々 → いろいろ |
| **接続詞** | 又 → また / 及び → および / 但し → ただし / 更に → さらに / 従って → したがって / 即ち → すなわち / 並びに → ならびに |
| **補助動詞・補助形容詞** | 〜て見る → 〜てみる / 〜て頂く → 〜ていただく / 〜て下さい → 〜てください / 〜て居る → 〜ている / 〜て来る → 〜てくる / 〜たく無い → 〜たくない / 〜て欲しい → 〜てほしい |
| **副助詞** | 等 → など / 迄 → まで / 程 → ほど / 位 → くらい |
| **連体詞** | 色んな → いろんな / 所謂 → いわゆる |
| **敬語表現** | 致します → いたします / 頂く → いただく / 下さい → ください |

判定 trick: 「**手書きでもこの漢字を書くか?**」を自問する。PC は無自覚に変換するため、書かないなら開く。具体物 vs 抽象概念で迷ったら、抽象側はひらく。

機械での完全な自動化は困難 (品詞の解析が必要)、上記表を sweep 基準として参照する。

出典: 共同通信社『記者ハンドブック 第 14 版』 (2022)、各種 Web ライティング規約 (テクニカルライティング業界共通)。

### 冗長表現の圧縮

| Before (冗長) | After (圧縮) |
|---|---|
| 〜することができる | 〜できる |
| 〜を行う | 〜する |
| 〜であると言えます | 〜である |
| 〜ということになる | 〜となる |
| 〜することが可能 | 〜できる |

出典: textlint-rule-ja-no-redundant-expression。

### 弱い表現の禁止 (技術文書本文)

技術文書 (DD / RCA / 報告書) の本文で曖昧語を使わない。断定できないなら「検証が必要」「未確認」と明示する。

| 禁止語 | 置換 |
|---|---|
| 〜かもしれない | 検証が必要 / 仮定: 〜 |
| 〜と思います | 〜と判断する / 未確認 |
| 〜と思われる | 〜と推定する (根拠を併記) |
| 〜可能性がある | 条件: 〜 / 発生率 X% |

既存の「断定語 warn-only」とは逆方向 — 断定語は過剰断定の抑制、弱い表現禁止は曖昧の排除。両方を併用する。出典: textlint-rule-ja-no-weak-phrase。

### 逆接「が」の連続禁止

1 文中に逆接の「〜が、〜が」が複数回出ない。逆接でない順接の「が」も連続で使わない (文の連結器として濫用しない)。出典: textlint-rule-no-doubled-conjunctive-particle-ga。

### 技術文書で `! ?` 不使用

PR / DD / RCA / Notion / commit message の本文 / 見出し / 箇条書きで `!` `！` `?` `？` を使わない。Slack chat は例外。出典: textlint-rule-preset-ja-technical-writing。

### 漢数字 / 算用数字の使い分け

- **算用数字**: 数量・数えられるもの (`3 件` / `10 人` / `5 GB`)
- **漢数字**: 熟語・慣用表現・概数・副詞 (`三位一体` / `数十年` / `一日中`)

混在は同一文書内で統一。出典: 文化庁「公用文作成の考え方」(2022)。

### 見出しは常体 / 体言止め統一

本文がですます調でも、heading は常体 / 体言止めが標準。本文常体なら heading は文末省略 (`〜する` / `〜の概要`)、本文ですます調でも heading は同じ。出典: textlint-rule-preset-JTF-style 1.1.2。

### 全角かっこ前後にスペースを置かない

- NG: `(例) 〜` / `〜 (補足) です`
- OK: `（例）〜` / `〜（補足）です`

英数字前後の半角 space rule と独立 (英数字には space あり、全角かっこには space なし)。出典: JTF-style 3.3。

## 避けるパターン (段落・文章全体の構造)

| パターン | 症状 | 対処 |
|----------|------|------|
| 抽象的なポエム | 名詞だけで構成、絵が浮かばない | 具体的な状況・行動・数値に置換 |
| 用語の羅列 | 専門用語が並ぶが各々の関係が不明 | 関係を1文で繋ぐか、1つに絞る |
| 過剰に丁寧な表現 | 「いただきますよう〜」「〜していただければ〜」 | 直接的な動詞文に短縮 |
| 余分な形容詞 | 「強力な」「革新的な」「柔軟な」が情報を持たない | 削除or数値 / 事例に置換 |
| 同義反復 | 同じ内容を言い換えて繰り返す | 1箇所に集約 |
| 中身のないまとめ | 「以上をまとめると重要だ」で何も結論してない | 1文の結論を書くか、まとめ自体を削除 |
| PR 文脈依存の表現 | 「DD §X-Y 参照」「incident (YYYY-MM-DD) で観測」等、半年〜1年後の読者に通じない社内文書参照・日付 | 必要情報は本文に書き起こし、社内URL・過去日付は削除する |
| 時限的マーカー (merge 後意味喪失) | 「本 PR で新設」「本 commit で削除」「先週合意した通り」「上述の通り」「直近の incident で」等、merge / 投稿後の読者に通じない時限文言 | 削除 or timeless 表現に置換 (「実装初期」「初版」等は OK)、状態完了マーカー (「対策反映済み」) は timeless として残す |
| 同 file 内 section 重複 (議論ポイント表 / 監視 / 採用理由 等) | 同論点が複数 section に分散、片方が要約 / 片方が詳細でも内容重複 | 詳細 section を残し、要約側は anchor link 1 行に圧縮。即統合できない場合は併存維持で別 task 化 |

## 出力前セルフチェック (6項目、5項目以上で合格)

```
[ ] TL;DR / 結論が冒頭 1-3 文 (「本稿では〜」削除済)
[ ] 「必須」「推奨」「重要」に 1 文の根拠併記
[ ] 初出専門用語に定義併記
[ ] 抽象語 (改善 / 最適化 / 効率化) の代わりに数字 or 事例
[ ] 読み手の次アクション (approve / 実行 / 質問) が末尾明示
[ ] `## 書く前の4問` (L27) の 4 問に答えてる
```

未達は修正してから出力。投稿 / commit / push前にAI自身が確認。

## 機械検出 grep (出力前 sweep)

self-check 6 項目通過後、残存 NG の機械検出を 1 行 grep で実施する。
chat draft 段階では hook block が発火しないため、自走 sweep を規範とする。

**一括検査**: `scripts/jp-textlint.sh <file>` で 連続漢字≥5 / 読点≥4 / 文長>100 / NG辞書 hit をまとめて検出できる (textlint 相当、手動起動)。pre-tool-use hook は外向き書込時に連続漢字・読点を warn-only で自動検出する (block はしない)。

### 基本 grep (略記・記号・jargon)

```bash
grep -nE '(M 行|K 行|→|外挿|諸元|逐次伝播|long TX|replica lag|disk I/O)' <draft>
```

### NG 辞書由来 grep (AI 定型語 / カタカナ造語禁止)

NG 語 canonical: [NG-DICTIONARY.md](NG-DICTIONARY.md)。全 key (AI定型語 / カタカナ造語禁止 / 難読漢語 / 弱い表現 / 冗長表現 / 非日常英語) の literal はそちら参照。hit したら該当行を書き換えてから出力する。

## 全体マップ — 各ルールがどう認知負荷を下げるか

ルールは「読み手が払うはずだった負荷を書き手が肩代わりする」ためにある。形だけのルール準拠はAI調に戻る。

| ルール | 認知負荷をどう下げるか |
|--------|---------------------|
| 4問 | 「誰のためか」を最初に絞り、読み手が自分事として読める |
| 媒体別構造 | 形が決まっていれば読み手は内容に集中できる |
| 文体分離 | documentは常体・主語必須・指示語禁止 → 読み返し不要 |
| 3変換 | 抽象語の解釈労力を書き手が引き受ける |
| 守る指針 | 段落ごとに役割を持たせ、見出しだけで流れが伝わる |
| セルフチェック | 文体崩れを機械的に検出、見落としを救う |

迷ったら「これは読み手の負荷をどう下げるか」を自問する。答えられないルール適用は形だけ。
