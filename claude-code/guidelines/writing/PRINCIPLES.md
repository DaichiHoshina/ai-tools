# 文章執筆の共通原則

ヒト向けドキュメント全般 (PR / Issue / Slack / Notion / DesignDoc / PRD / RCA / コメント) を書くときの共通原則。各ファイル ([commit-message](commit-message.md) / [pr-description](pr-description.md) / [external-post](external-post.md) / [long-form-doc](long-form-doc.md) / [design-doc-protocol](design-doc-protocol.md)) は本ファイルを起点に「適用先別の追加ルール」を扱う。

## 最優先 — 読み手の認知負荷を下げる

ドキュメントは記憶ダンプでなく、判断 / 行動を助ける道具。すべてのルールはこの目的から派生する。

- 1 回読めば理解できる文章を目指す
- 読み手に「で、つまり何？」と思わせない
- 「賢そう」でなく「伝わる」を優先する
- 初見の人が途中で止まらないこと

迷ったらこの 4 つに戻る。判断の優先順は「ルール準拠」より「読み手の負荷」。

## 書く前の 4 問

抽象的な答え不可。答えずに書くと AI 文体になる。

1. **読み手は具体的に誰か** (×「エンジニア」○「新 join の senior backend」「レビュアー Ayano さん」)
2. **読後に何を決めさせる / やらせるか** (×「理解させる」○「approve を決めさせる」「migration を実行させる」)
3. **主張を裏付ける数字 or 事例** (×「大幅改善」○「p99 1.2s → 320ms (-73%)」)
4. **「なぜ？」と聞かれたら** (×「〜が必須」○「localStorage に token 置かない、XSS で抜かれる」)

## 守る指針 (7 原則)

1. **用語を並べるだけで説明しない**: 専門用語の羅列は内容ゼロ。1 用語ごとに「何のために・どう動くか」を 1 文添える
2. **暗黙知を前提にしない**: 「いつもの」「ご存知の通り」を使ったら、その内容を 1 文で書く
3. **次の疑問を先回り**: 1 段落書いたら「読み手が次に何を聞きたいか」を想像し、次段落に答えを置く
4. **見出しだけで流れが分かる**: 見出しはラベル (「アーキテクチャ」) でなく主張 (「読み書きを分離して負荷を分散する」)
5. **段落に役割を持たせる**: 各段落は「背景 / 理由 / 具体例 / 結論 / 注意点」のどれかに該当。役割が曖昧なら削除 or 分割
6. **抽象説明の直後に具体例**: 抽象名詞や原則を書いたら、次の 1-2 文で具体例 (数値・事例・コード断片)
7. **「賢そう」より「伝わる」**: 平易で短い文を選ぶ。読み手は実務判断のために読む

## 媒体別構造

### 短文 (issue / PR コメント / Slack / Notion コメント / ticket)

**PREP 3 点構造**:

1. 結論: 読み手に何を判断 / 実行させるか (1 行)
2. 理由: 現象 / 影響 / 原因
3. 次アクション: 担当 / 期限 / 不明点

- 本文 400 字前後 (30 秒で読める)、title 80 字前後
- 詳細ログ・スタックトレースは `<details>` 折りたたみ or 別ファイルリンク
- 詳細は [external-post.md](external-post.md) 参照

### 長文 (Design Doc / PRD / RCA / Notion ページ)

- **TL;DR 冒頭 1-3 文で結論**。「本稿では〜について述べる」系導入は削除
- 800-1500 字 / ページ、2000 字超なら分割
- 1 セクション = 地の文 3-5 割 + 箇条書き / 表 5-7 割
- 箇条書きは 3-5 項目。7 超は表 or 段落分割
- コードブロックは 5 行以内
- 詳細は [long-form-doc.md](long-form-doc.md) 参照

### PR description

4 セクション固定: **Why** / **What changed** (数字・事例) / **Testing** / **Review focus**。詳細は [pr-description.md](pr-description.md) 参照。

### DesignDoc

12 セクション or 軽量 5 節。詳細は [design-doc-protocol.md](design-doc-protocol.md) 参照。

## chat と document で文体を分ける

| 場面 | 方針 |
|------|------|
| Chat 応答 (genshijin モード等) | 体言止め可、助詞省略可、結論先出し、短く速く |
| ヒト向け document | 文として完結 (〜する / 〜した)、主語明示、地の文と箇条書き混在、常体統一 |

**document 基本ルール**:

- 主語必須 (「担当者は〜」「ユーザーは〜」)
- 指示語禁止 (「これ」「それ」「上記」「前述」→ 具体名)
- 略語禁止 (初出はフルスペル)
- 「適宜」「状況に応じて」禁止 → IF-THEN 形式で明文化
- 日付 YYYY-MM-DD、英数字半角

## AI 臭を消す 3 変換

### (a) 抽象語 → 数字 / 事例

| NG | OK |
|----|----|
| レスポンス速度を大幅改善 | p99 1.2s → 320ms (-73%)。Redis cache + N+1 削除 |
| エラー処理を強化 | 5xx 日次 120 件 → 8 件。`context.WithTimeout` を全 handler に追加 |
| 効率的に作成可能 | md 直接投稿できる。手動コピペ廃止が目的 |

### (b) 評価語 → 根拠 1 文併記

「適切な / 最適な / 改善 / 重要 / 必須 / 推奨 / 一般的に / 強化 / 向上 / 活用」を書くなら**直後に 1 文で根拠** (なぜ・事例・数字)。

| NG | OK |
|----|----|
| セッショントークンの適切な管理が必須 | session token は cookie `httpOnly` に。localStorage は XSS 経由で盗まれ、token に expiry しかないため全アカウント乗っ取り可 |

### (c) 難語 → 削除でなく定義併記

初出で 1 文の定義を併記。専門用語を削ると正確さを失う。

| NG | OK |
|----|----|
| idempotency を担保 | idempotency (同リクエストを 2 回叩いても結果不変) を担保。retry で二重注文発生しない |
| CQRS で整合性 | CQRS (読み書きを別モデル分離) で整合性。read は replica から返せて scale 可 |

### NG 辞書 (削除対象)

**AI 定型語**: 効果的に / 効率的に / シームレスに / 直感的に / 革新的な / 素晴らしい / 強力な / より良い / 〜を実現します / 〜を提供します / 〜を可能にします / 〜することができます / ご紹介します / ご覧ください / 〜いただけます / まず〜しましょう / 重要なポイント / 注目すべき点 / 本機能は / 本ドキュメントは / 本記事では / 本稿では〜について述べる

**硬い文語 (柔らかく)**: 〜である / 〜であった / 〜されている / 〜となる / 〜に基づき / 〜に関して / 当該 / 以下に示す

## 段落レベル Before/After

### ❌ AI 調 (抽象ポエム + 用語羅列)

> 可観測性向上を目的として疎結合なイベント駆動アーキテクチャを導入し、スケーラビリティを担保する。これにより、システム全体の柔軟性と保守性が大幅に改善される。

問題: 抽象名詞の羅列で絵が浮かばない / 何が起きていて何を変えるか不明 / 「大幅に改善」の根拠ゼロ。

### ⭕ 自然な実務文 (具体状況 → 具体解決)

> 注文処理が重くなったときに、一部の処理だけ遅延できるようにしたい。そのため、メール送信や通知処理を別の非同期処理に分離する。本処理 (在庫引当・決済) の応答時間を p95 500ms 以内に保つことが目的。

抽象→具体への変換は機械的にできる: 抽象名詞を「いま起きている具体的な事象」「これから取る具体的な行動」に分解する。

## 避けるパターン (段落・文章全体の構造)

| パターン | 症状 | 対処 |
|----------|------|------|
| 抽象的なポエム | 名詞だけで構成、絵が浮かばない | 具体的な状況・行動・数値に置換 |
| 用語の羅列 | 専門用語が並ぶが各々の関係が不明 | 関係を 1 文で繋ぐか、1 つに絞る |
| 過剰に丁寧な表現 | 「いただきますよう〜」「〜していただければ〜」 | 直接的な動詞文に短縮 |
| 余分な形容詞 | 「強力な」「革新的な」「柔軟な」が情報を持たない | 削除 or 数値 / 事例に置換 |
| 同義反復 | 同じ内容を言い換えて繰り返す | 1 箇所に集約 |
| 中身のないまとめ | 「以上をまとめると重要だ」で何も結論してない | 1 文の結論を書くか、まとめ自体を削除 |

## 出力前セルフチェック (6 項目、5 項目以上で合格)

```
[ ] TL;DR / 結論が冒頭 1-3 文 (「本稿では〜」削除済)
[ ] 「必須」「推奨」「重要」に 1 文の根拠併記
[ ] 初出専門用語に定義併記
[ ] 抽象語 (改善 / 最適化 / 効率化) の代わりに数字 or 事例
[ ] 読み手の次アクション (approve / 実行 / 質問) が末尾明示
[ ] 4 問 (読み手 / ゴール / 数字 / なぜ) に答えてる
```

未達は修正してから出力。投稿 / commit / push 前に AI 自身が確認。

## 全体マップ — 各ルールがどう認知負荷を下げるか

ルールは「読み手が払うはずだった負荷を書き手が肩代わりする」ためにある。形だけのルール準拠は AI 調に戻る。

| ルール | 認知負荷をどう下げるか |
|--------|---------------------|
| 4 問 | 「誰のためか」を最初に絞り、読み手が自分事として読める |
| 媒体別構造 | 形が決まっていれば読み手は内容に集中できる |
| 文体分離 | document は常体・主語必須・指示語禁止 → 読み返し不要 |
| 3 変換 | 抽象語の解釈労力を書き手が引き受ける |
| 守る指針 | 段落ごとに役割を持たせ、見出しだけで流れが伝わる |
| セルフチェック | 文体崩れを機械的に検出、見落としを救う |

迷ったら「これは読み手の負荷をどう下げるか」を自問する。答えられないルール適用は形だけ。
