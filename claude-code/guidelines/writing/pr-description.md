# PR本文 + レビュー応答

## レビューコメント添字 (推奨)

| 添字 | 意味 |
|---|---|
| `must` | 修正必須 |
| `imo` | 任意・意見 |
| `nits` | 細かい指摘 |
| `q` | 質問 |

プロジェクト規約が存在しない場合の汎用 default として使う (プロジェクト規約が違えばそちらに従う)。

## 本文の書き方

- **箇条書きファースト**: 散文段落を避け、bulletで構造化。レビュアーがscanできる形に
- **階層で関係を表現**: 親子・因果・包含関係はインデント / サブbullet。散文で接続詞を重ねない
- **構造（場所）で束ねる**: 変更を「ファイル / モジュール / レイヤー」単位でグループ化。**「what / why / how」など抽象観点でsectionを割らない**
- **section重複を避ける**: 同じ事実は1ヶ所のみ。背景・原因・影響範囲で同内容を繰り返さない
- **必要な識別子は書く**: クラス名・設定キー・関数名は変更スコープ明示に必要なら記載OK。逆にdiffから自明な変数名の羅列・実装詳細は省く
- **長さは内容に従う**: 自明な変更は数行、設計判断を含む変更は原因や代替案も記述
- **Web 可読性**: GitHub / GitLab PRは scan 前提。1 文 60 字 / 主張型 heading / 太字 scan 化 — 詳細 `PRINCIPLES.md` `## Web 可読性`

## テンプレ (ai-tools canonical、全 repo 優先)

```
## 背景
## Related Issue
## 実装概要
## 影響範囲
## 動作確認エビデンス
## 動作確認手順
## 備考
```

- ai-tools canonical を全 repo で project template (`.github/pull_request_template.md` 等) より優先する
- 関連Issueは必ず紐付け (`Closes #XXX`)
- 別repoのIssueは `org/repo#XXX` 形式
- project template に label / checkbox 等の固有要素 (例: `operation check` label / `- [ ] マニュアルテスト実施可否`) があれば該当 section に転記する

## 禁止

- 「Generated with X」「AIが生成」等のフッター
- テーブル形式のファイル一覧 (diffで読める)
- 過剰絵文字・絵文字主体のsectionヘッダ
- commit message の貼付け直し (`git log` で読める。本文は背景・影響・検証に集中)
- **経過メモ (時系列の変遷記述)**: 「当初 X で実装したが Y に変更」「レビュー指摘で Z に修正」等の開発中の変遷は書かない。PR body は**現在の diff の事実のみ**を記述する (最終形の背景 / 影響 / 検証)。変遷は commit history とレビュースレッドが保持する。設計を途中で変えた理由が読み手に必要な場合のみ、「Z を採用した。X は〜のため不採用」と**現在形の採用理由**として 1-2 行で書く

## レビュー応答 (PR修正時)

| ルール | 詳細 |
|---|---|
| Resolve主体 | **レビュワー** がresolveにする。実装者は基本resolveしない |
| 修正リンク | コメントへの修正対応には必ず**修正コミットのリンク**も添付 |
| commit粒度 | **1コメント = 1 commit** で整理 (force-push可)。textlint / lint修正単独commitは作らず対応commitに合流 |
| 返信例コピー | ユーザーが返信例を要求した場合、生成 + クリップボードコピーまで |

## PR body 追記ルール

- **後付け H2を並べない**: 「追加対策」「最新ステータス」等の時系列追記は既存セクションに統合する
- **箇条書きは 1 行に結論**: 詳細値 (commit hash / 関数名 / 数値) は同行内 `()` に添える (`- キャッシュ TTL を 60s に短縮 (修正: a1b2c3d, 関数: `setCacheTTL`)`)
- **「補足」「最新ステータス」の位置**: CodeRabbit Summaryの前に置く (auto-gen 領域の後ろは読まれない)

## 動作確認セクションの構成

- **1セクションに統合**: 同種の検証を `## 動作確認` 1セクションに集約し、`### ローカル` / `### dev リハーサル` / `### 負荷試験` の小見出しで時系列分離する
- **並列に何本も生やさない**: 検証セクションを `## 動作確認 (ローカル)` / `## 動作確認 (dev)` のように H2 並列で並べると、後付け H2と区別がつかなくなる
- **時系列追記は既存小見出しに統合**: 「再検証」「追加検証」等を新規H3で並べず、既存小見出しに統合する
- **手順は 3 項目上限**: scan 前提で 4 項目以上に膨らむと読まれない。超過分は別 doc / dashboard URL / log 共有 link に逃がし、PR 本文には要約 + 参照のみ残す

## 外部共有不可 path / URLを貼らない

- **ローカル絶対 pathは貼らない**: `/Users/<name>/...` 等の絶対 pathは repo 外の読み手が解決できない
- **gitignored / 外部不可視 pathは貼らない**: `.local/` / 個人 dir 配下 fileの pathは PR reviewerが辿れない
- **必要な内容は本文に書き起こす**: 外部共有可能な場所 (Notion 共有page / dashboard URL / PR 本文直書き) に転載する
- **path 貼る時の判定基準**: 「reviewerがcloneした repo 内で解決可能か」を確認する。Noなら本文展開 or 共有可能 URLに置き換え

## レビュアー視点のチェック

- 1つのsectionに同じ事実が出てきていないか
- 「ファイル単位」と「観点単位」のsectionが混在していないか
- 動作確認手順を見て、第三者が再現できるか
