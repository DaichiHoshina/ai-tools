# PR 本文 + レビュー応答

## レビューコメント添字 (推奨)

| 添字 | 意味 |
|---|---|
| `must` | 修正必須 |
| `imo` | 任意・意見 |
| `nits` | 細かい指摘 |
| `q` | 質問 |

プロジェクトで規約が違えばそちらに従う。本ガイドは「無いとき」の汎用 default。

## 本文の書き方

- **箇条書きファースト**: 散文段落を避け、bullet で構造化。レビュアーが scan できる形に
- **階層で関係を表現**: 親子・因果・包含関係はインデント / サブ bullet。散文で接続詞を重ねない
- **構造（場所）で束ねる**: 変更を「ファイル / モジュール / レイヤー」単位でグループ化。**「what / why / how」など抽象観点で section を割らない**
- **section 重複を避ける**: 同じ事実は 1 ヶ所のみ。背景・原因・影響範囲で同内容を繰り返さない
- **必要な識別子は書く**: クラス名・設定キー・関数名は変更スコープ明示に必要なら記載 OK。逆に diff から自明な変数名の羅列・実装詳細は省く
- **長さは内容に従う**: 自明な変更は数行、設計判断を含む変更は原因や代替案も記述

## テンプレ (プロジェクトで指定があればそちら優先)

```
## 背景
## Related Issue
## 実装概要
## 影響範囲
## 動作確認エビデンス
## 動作確認手順
## 備考
```

- 関連 Issue は必ず紐付け (`Closes #XXX`)
- 別 repo の Issue は `org/repo#XXX` 形式

## 禁止

- 「Generated with X」「AI が生成」等のフッター
- テーブル形式のファイル一覧 (diff で読める)
- 過剰絵文字・絵文字主体の section ヘッダ

## レビュー応答 (PR 修正時)

| ルール | 詳細 |
|---|---|
| Resolve 主体 | **レビュワー** が resolve にする。実装者は基本 resolve しない |
| 修正リンク | コメントへの修正対応には必ず**修正コミットのリンク**も添付 |
| commit 粒度 | **1 コメント = 1 commit** で整理 (force-push 可)。textlint / lint 修正単独 commit は作らず対応 commit に合流 |
| 返信例コピー | ユーザーが返信例を要求した場合、生成 + クリップボードコピーまで |

## レビュアー視点のチェック

- 1 つの section に同じ事実が出てきていないか
- 「ファイル単位」と「観点単位」の section が混在していないか
- 動作確認手順を見て、第三者が再現できるか
