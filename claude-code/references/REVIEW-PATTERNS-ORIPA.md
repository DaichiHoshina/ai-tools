# レビュー指摘パターン集（オリパ）

rinchsanのPRレビューから抽出した頻出パターン。オリパ関連リポジトリで作業時に参照。

## Go（バックエンド）

### 1. 構造体の使い回し禁止

→ `golang.md` 意味的型分離ルール参照

### 2. 過剰な共通化を避ける

呼び出し元で分岐が済んでいるなら、内部でさらに分岐するヘルパーは不要。直接呼ぶほうがシンプル。ただしSQLの共通化（WHERE条件だけ違う場合）は積極的にやる。

### 3. 不要なInputは作らない

Usecaseに渡すパラメータがなければInput構造体を作らず、引数なしにする。

### 4. ドメインモデルの変更はmodelのメソッドで

Usecaseでフィールドを直接書き換えない。`SetXxx()` メソッドを実装する。関連フィールドの同時変更が必要になったとき、Usecase側の修正が不要になる。

### 5. テスト必須・gomock使用

- Usecaseにはテストを書く。テストなしはNG
- 自前stubではなくgomockを使う
- テストの期待値を簡略化してアサーションを弱くしない
- 新しいロジック追加時は、そのロジックのテストケースも追加

### 6. 命名は周囲と合わせる

Router関数名等は周囲の既存コードに合わせる。

### 7. 未使用コードを混入させない

- そのPRの差分で使わないinterfaceメソッド・関数は含めない
- ブランチがごちゃついて後続PRのコードが混ざるケースに注意

### 8. エラーステータスコードの選択

- サーバー側のデータ不整合（view変換失敗等）→ 500。400にしない
- リソースが見つからない → NotFound。BadRequestにしない

### 9. SELECT * 禁止

必要なカラムのみ指定する。Linter（select_star_guard）が動作している場合はそれに従う。

### 10. nullable.Nullable の使用

→ `golang.md` エンティティ・Nullable 参照

## フロントエンド（Vue）

### 11. v-modelで双方向バインディング

`@change` + ハンドラで値を書き換えるより `v-model` で連動させる。

### 12. 常にtrueな条件を書かない

API呼び出しの戻り値等、常にtruthyな値の条件分岐は不要。

### 13. API呼び出しは並列化

複数回のAPI呼び出しは `Promise.all` で並行実行。直列だとユーザーが待つ。

### 14. 部分失敗時のUI整合性

複数APIのうち一部だけ失敗した場合、DBとUIの不整合が起きる。失敗時はリフェッチするか、バッチAPIでオール・オア・ナッシングにする。

### 15. ローディング中のちらつき防止

データ切替→API取得完了までの間に「0件」表示が出ないよう、loading状態を管理する。

## マイグレーション

### 16. migration番号の衝突

mainに入っている最新番号を確認してから番号を振る。マージ直前に再確認。

### 17. down migrationでDEFAULT句を明示

既存migrationに倣い、MODIFY COLUMNではDEFAULT NULL等を明示する。

## 設計判断

### 18. APIの分割・統合を検討する

新しいAPIを作る前に、既存APIに機能を追加するほうが自然でないか検討する。Design Docの段階で議論しておく。
