# 文単位の品質規約 (詳細)

`guidelines/writing/PRINCIPLES.md` の `## 文単位の品質規約` の詳細層。段落 / 媒体構造は PRINCIPLES.md 本体が扱い、本 file は 1 文単位の品質を定量規範化する。textlint preset (ja-technical-writing / JTF-style) の運用知見を取り込む。深い書き直し / `/jp-writing` 実行時に参照する (毎回 auto-load 不要のため reference へ分離)。

## 文長 (媒体別 2 段)

- 技術文書本文 (DD / RCA / long-form-doc): 1 文 100 字以内
- web / 短文 (PR / Slack / Notion): 1 文 60 字以内 (PRINCIPLES.md `## Web 可読性` で既出)

文単位の上限は媒体で切替する。100 字超は分割の合図とする。

## 読点 3 個まで

1 文中の読点 (`、`) は 3 個まで。超えたら文分割する。読点 4+ は読者の認知負荷が急上昇する。出典: textlint-rule-preset-ja-technical-writing (max-ten rule)。

## 読点は係り受けの境界に打つ

読点の個数上限 (前節) とは別に、打つ位置は係り受けで決める。息継ぎや見た目のリズム目的では打たず、係り先を固定する記号として使う。

1. 長い修飾語が 2 つ以上並ぶとき、その境界に打つ
2. 語順の原則 (長い修飾語が先) を崩して短い語を前に出したとき、その直後に打つ

- NG: `渡辺刑事は血まみれになって逃げた賊を追った。` (血まみれなのが刑事か賊か 2 通りに読める)
- OK: `渡辺刑事は、血まみれになって逃げた賊を追った。`

出典: 本多勝一『日本語の作文技術』読点の 2 原則。

## 修飾語の語順 4 原則

係り先の誤読は読点でなく語順で防ぐのが先決だ。

1. **長い修飾語を先に**、短い修飾語を被修飾語の直前に置く
2. **述語を含む節を先に**、述語を含まない句を後に置く
   - NG: `厚手の白い横線の引かれた紙` (「白い横線」と誤読する)
   - OK: `横線の引かれた厚手の白い紙`
3. **大状況 (時期・環境) から小状況 (個別対象) へ**の順に置く
   - NG: `決済 API で今朝の deploy 後に error が増えた`
   - OK: `今朝の deploy 後に決済 API で error が増えた`
4. **前置修飾節を入れ子にしない** (逆茂木型の禁止)。長い修飾節が 2 つ以上要るなら文を分割する
   - NG: `昨夜 deploy した際に発生した、retry 処理が二重に走る原因となった bug を修正した`
   - OK: `bug を修正した。この bug は昨夜の deploy で発生し、retry 処理を二重に走らせていた`

出典: 本多勝一『日本語の作文技術』/ 文化庁「公用文作成の考え方」(2022) / 木下是雄『理科系の作文技術』(逆茂木型)。

## 同一助詞の連続回避

1 文の中で同じ助詞 (「の」「に」「も」「て」) を短い間隔で 3 回以上続けない。動詞化か語順変更で言い換える。隣接重複 (「のの」) だけでなく、離れた連鎖も対象とする。

- NG: `本年の当課の取組の中心は AI 導入だ`
- OK: `本年、当課が中心的に取り組んでいるのは AI 導入だ`

機械検出: `scripts/jp-textlint.sh` が「の」3 連の鎖を検出する。出典: 文化庁「公用文作成の考え方」/ textlint-rule-no-doubled-joshi。

## 受身は主体が書けるなら能動で書く

受身 (`〜される` / `〜られる`) をむやみに使わない。動作の主体を書けるなら能動で書き、1 文に受身を 2 回以上重ねない。受身の連続は文構造を複雑にし、誰がやったかの所在を曖昧にする。

- NG: `検証が実施され、問題が確認され、修正が行われた`
- OK: `test で検証して問題を 2 件見つけ、修正した`

例外 (UI 操作説明): 読み手の操作は能動 (`[保存] を押す`)、システム側の結果は受動 (`データが保存される`) で書き分け、態そのもので主体を示す。

機械検出: `scripts/jp-textlint.sh` が 1 文内の受身 2 連を検出する。出典: 文化庁「公用文作成の考え方」/ 木下是雄『理科系の作文技術』/ SmartHR Design System。

## ら抜き言葉禁止

ら抜き (見れる / 食べれる / 来れる) を使わない。可能の意味は「〜できる」への言い換えが最も明確だ。

- NG: `この画面から log が見れる`
- OK: `この画面で log を確認できる`

出典: textlint-rule-no-dropping-the-ra。

## 連続漢字 4 文字上限

漢字 5 文字以上の連続は読解負荷が高い。固有名詞・技術用語 (e.g. `Opus`、`hook`) は例外、漢語連結のみ対象。

| Before (連続漢字) | After (助詞挿入 / 訓読み開く / 動詞化) |
|---|---|
| `利用者認証処理管理` | `利用者の認証処理を管理` |
| `構造的変化傾向分析` | `構造的な変化傾向の分析` |
| `段階的強化` | `段階的に強化` |
| `逐次化検出` | `逐次に並ぶ状態を検出` |
| `絶対禁止徹底` | `禁止を徹底する` |

機械検出: `grep -oE '[一-龯]{5,}'` で外向き prose を sweep する。

出典: textlint-rule-preset-ja-technical-writing (max-kanji-continuous-len rule) を 2026-06-08 に 6→4 へ厳格化。

## ひらく漢字 7 品詞 (ひらがな表記推奨)

外向き prose では下記 7 品詞をひらがな表記する。記者ハンドブック (共同通信社) 業界標準に準拠。具体物・物体・時刻そのもの等の本来の意味で使う場合は漢字維持、抽象・概念・形式語の場合のみひらく。

| 品詞 | NG (とじる) → OK (ひらく) |
|---|---|
| 形式名詞 | する事 → すること / する時 → するとき (場合の意) / する為 → するため / する上で → するうえで / の様 → のよう / の通り → のとおり / する所 → するところ |
| 副詞 | 既に → すでに / 例えば → たとえば / 是非 → ぜひ / 沢山 → たくさん / 全て → すべて / 様々 → さまざま / 暫く → しばらく / 大分 → だいぶ / 恐らく → おそらく / 色々 → いろいろ |
| 接続詞 | 又 → また / 及び → および / 但し → ただし / 更に → さらに / 従って → したがって / 即ち → すなわち / 並びに → ならびに |
| 補助動詞・補助形容詞 | 〜て見る → 〜てみる / 〜て頂く → 〜ていただく / 〜て下さい → 〜てください / 〜て居る → 〜ている / 〜て来る → 〜てくる / 〜たく無い → 〜たくない / 〜て欲しい → 〜てほしい |
| 副助詞 | 等 → など / 迄 → まで / 程 → ほど / 位 → くらい |
| 連体詞 | 色んな → いろんな / 所謂 → いわゆる |
| 敬語表現 | 致します → いたします / 頂く → いただく / 下さい → ください |

判定 trick: 「手書きでもこの漢字を書くか?」を自問する。PC は無自覚に変換するため、書かないなら開く。具体物 vs 抽象概念で迷ったら、抽象側はひらく。

機械での完全な自動化は困難なため (品詞の解析が必要となる)、書き手は「ひらく漢字 7 品詞」表を sweep 基準として参照する。

出典: 共同通信社『記者ハンドブック 第 14 版』 (2022)、各種 Web ライティング規約 (テクニカルライティング業界共通)。

## 冗長表現の圧縮

| Before (冗長) | After (圧縮) |
|---|---|
| 〜することができる | 〜できる |
| 〜を行う | 〜する |
| 〜であると言えます | 〜である |
| 〜ということになる | 〜となる |
| 〜することが可能 | 〜できる |

出典: textlint-rule-ja-no-redundant-expression。

## 弱い表現の禁止 (技術文書本文)

技術文書 (DD / RCA / 報告書) の本文で曖昧語を使わない。断定できないなら「検証が必要」「未確認」と明示する。

| 禁止語 | 置換 |
|---|---|
| 〜かもしれない | 検証が必要 / 仮定: 〜 |
| 〜と思います | 〜と判断する / 未確認 |
| 〜と思われる | 〜と推定する (根拠を併記) |
| 〜可能性がある | 条件: 〜 / 発生率 X% |

既存の「断定語 warn-only」とは逆方向 — 断定語は過剰断定の抑制、弱い表現禁止は曖昧の排除。両方を併用する。出典: textlint-rule-ja-no-weak-phrase。

## 逆接「が」の連続禁止

1 文中に逆接の「〜が、〜が」が複数回出ない。逆接でない順接の「が」も連続で使わない (文の連結器として濫用しない)。出典: textlint-rule-no-doubled-conjunctive-particle-ga。

## 技術文書で `! ?` 不使用

PR / DD / RCA / Notion / commit message の本文 / 見出し / 箇条書きで `!` `！` `?` `？` を使わない。Slack chat は例外。出典: textlint-rule-preset-ja-technical-writing。

## 漢数字 / 算用数字の使い分け

- 算用数字: 数量・数えられるもの (`3 件` / `10 人` / `5 GB`)
- 漢数字: 熟語・慣用表現・概数・副詞 (`三位一体` / `数十年` / `一日中`)

混在は同一文書内で統一する。4 桁以上の数値は 3 桁ごとに半角カンマで区切る (`1,048,576`)。年号・ID・code 内は桁区切りの対象外。出典: 文化庁「公用文作成の考え方」(2022) / JTF 日本語標準スタイルガイド。

## 見出しは常体 / 体言止め統一

本文がですます調でも、heading は常体 / 体言止めが標準。本文常体なら heading は文末省略 (`〜する` / `〜の概要`)、本文ですます調でも heading は同じ。出典: textlint-rule-preset-JTF-style 1.1.2。

## 全角かっこ前後にスペースを置かない

- NG: `(例) 〜` / `〜 (補足) です`
- OK: `（例）〜` / `〜（補足）です`

英数字前後の半角 space rule と独立 (英数字には space あり、全角かっこには space なし)。出典: JTF-style 3.3。
