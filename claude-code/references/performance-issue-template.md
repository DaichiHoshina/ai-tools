# パフォーマンス改善 issue の進め方テンプレート

パフォーマンス改善・負荷対策を伴う機能開発で issue に何を残すかのテンプレート。Go の `pprof` を例にしているが、計測→分析→段階改善→負荷試験のフローは言語非依存。

## issue への記載フロー

### Phase 1: 検討・情報収集

issue コメントに以下のリンクを貼る。別途資料を作成してから着手。

```markdown
検討資料
{設計メモ・ドラフト URL}

過去の関連負荷試験記録
{Datadog Notebook / Grafana / 内部 wiki URL}

手法検討スレッド
{チャットスレッド URL}
```

**ポイント**:

- 過去に類似の負荷試験・調査があれば必ず探す
- 検討段階の情報も issue に集約する（後から辿れるように）

### Phase 2: ベンチマーク基盤の整備

改善に着手する前に計測環境を整え、issue に実行方法を記載。

```markdown
## ベンチマーク実行コマンド

{コマンド}

### プロファイルを取得する場合

{cpuprofile / blockprofile / trace 付きコマンド}

## ベンチマークテストのコード

#{関連 PR 番号} の変更を適用したコードを使用。

### ベンチマークの見方に関する注意

{何を計測しているか / ローカルでの計測である旨 / 絶対値ではなく相対比較で見る旨}
```

**ポイント**:

- 計測条件・前提を明記（並列数、delay、マシンスペック）
- 「何の数値か」を必ず説明（1 回の操作ではなく N 並列 × M 件の合計時間、等）
- 再現可能なコマンドを記載

### Phase 3: 改善前の状態を記録

```markdown
## 改善実施前のベンチマーク

{ベンチマーク結果}

- 生ログ: [bench-00-before-improvement.log](添付)

### 改善前時点の状況整理

#### 全体所感

- {主要なボトルネック}

#### {条件別}分析

- {条件A}: {観察結果}
- {条件B}: {観察結果}
```

**ポイント**:

- 生ログは添付ファイルで貼る（コメント本文は計測結果行のみ）
- 数値だけでなく「所感」「分析」を必ず書く

### Phase 4: pprof 分析（Go 例）

Go 標準の `pprof` でプロファイルを取得し、ボトルネックを特定する。他言語でも同等プロファイラ（Python: cProfile, Node: --prof, Ruby: stackprof 等）に読み替え。

#### pprof 取得コマンド

```bash
# cpuprofile + blockprofile + trace を同時取得
go test -tags serial \
  -bench={ベンチマーク名} -benchmem -benchtime=1x \
  -cpuprofile /tmp/{名前}.cpu.pprof \
  -blockprofile /tmp/{名前}.block.pprof \
  -trace /tmp/{名前}.trace \
  -run='^$' ./{パッケージ}/
```

#### 分析コマンド

```bash
# CPU: どの関数が CPU 時間を消費しているか
go tool pprof -top -cum {file}.cpu.pprof    # cumulative 順（呼び出し元込み）
go tool pprof -top -flat {file}.cpu.pprof   # flat 順（その関数自身の消費）

# Block: どこで goroutine がブロックしているか
go tool pprof -top {file}.block.pprof
```

#### 判断基準

| CPU サンプル率 | 意味 | 次のアクション |
|---|---|---|
| 高い（>20%） | CPU bound | アプリコードの最適化（アルゴリズム改善等） |
| 低い（<10%） | I/O bound | DB round-trip 削減、バルク化、クエリ最適化 |

| flat top の上位 | 意味 |
|---|---|
| syscall / runtime | DB I/O 待ち中心。CPU 最適化余地なし |
| アプリコード関数 | その関数がホットスポット。コード改善対象 |

#### issue コメント形式

```markdown
## プロファイル分析（{方式名} {規模}）

### 計測条件

- ベンチマーク: `{ベンチマーク名}`
- 結果: `{時間} / {メモリ} / {allocs}`
- マシン: {CPU} / Docker DB {バージョン}

<details>
<summary>CPU Profile 分析</summary>

{flat top 表 + cumulative top + 所見}

</details>

<details>
<summary>Block Profile 分析</summary>

{待ち時間の内訳 + 所見}

</details>

### 結論

{CPU bound / I/O bound の判定、ボトルネックの特定、次のアクション}
```

**ポイント**:

- **改善前後で 2 回取得**する。改善前で主犯特定→改善後で残課題特定
- `<details>` で折りたたむ（長くなるため）
- AI 分析を使う場合はプロンプトも記載（再現性のため）
- 分析後、次のアクションを 1 行コメントで宣言

### Phase 5: 段階的改善と計測

**1 改善 = 1 コメント**で記録。改善は小さく分割し、各段階で効果を計測。

```markdown
## {改善カテゴリ}

- [x] {対象テーブル/処理1}
- [x] {対象テーブル/処理2}
- [ ] {未着手}

### 1st: {具体的な改善内容}

改善後のベンチマーク

{結果}

{分析（前回との比較、条件別の変化、副次的影響）}

生ログ:

- [bench-01-{改善名}.log](添付)

### 2nd: {次の改善内容}

{同様のフォーマット}

## 所感

{ここまでの成果と次のアクションの判断}
```

**ポイント**:

- 改善を一気にやらない。1 つずつ計測して効果を確認
- 各段階で「効いたか/効かなかったか」を明記
- 所感で次のスコープ判断を記載（別 issue に送る判断も含む）

### Phase 6: スコープ判断・設計ドキュメント

改善途中で得た知見から、スコープの見直しや将来の設計提案を行う。

```markdown
{施策名}について整理。

- {別 issue との関係}
- {後方互換性の考慮}
- {現時点の改修でどこまでカバーできるか}

{スコープ判断の結論}

<details>
<summary>DesignDoc 素案</summary>

### {設計名}

{テーブル設計・クエリ設計・既存との互換性}

#### 採用しなかった案

- {却下案と理由}

</details>
```

**ポイント**:

- 「今やること」と「次の issue に送ること」を明確に分ける
- 将来の設計提案も issue に残す（DesignDoc 素案として）
- 判断根拠を書く（「ベンチマークからこの程度の性能が出る見込み」等）

### Phase 7: 負荷試験

```markdown
## 負荷試験の結果

{Datadog Notebook / Grafana ダッシュボード URL}
```

**ポイント**:

- 負荷試験結果はモニタリングツールに記録し、issue にはリンクのみ
- ローカルベンチマークとは別に、開発環境等での負荷試験を実施

### Phase 8: リリースタスク

```markdown
## リリースまでのタスク

- DesignDoc
  - [ ] 記述 {DesignDoc PR URL}
  - [ ] レビュー
- Backend API
  - [ ] {タスク1}（{見積}） #{PR 番号}
    - [ ] レビュー
  - [ ] {タスク2}（{見積}） #{PR 番号}
    - [ ] レビュー
- Frontend
  - [ ] {タスク}（{見積}） #{PR 番号}
    - [ ] レビュー
- テスト
  - [ ] 開発環境
- リリース
  - [ ] Backend
  - [ ] Frontend

### 備考

- DB スキーマ変更: {あり/なし}
- 後方互換性: {対応内容}
```

**ポイント**:

- 各タスクに PR 番号を紐付け
- 見積を記載
- レビューもチェック項目に含める
- 備考でスキーマ変更・後方互換の有無を明記
