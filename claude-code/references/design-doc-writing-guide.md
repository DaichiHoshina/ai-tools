# DesignDoc 書き方ガイド

`design-doc-scope-guide.md`（粒度・テンプレ選択）の補完。書き直しの実戦ノウハウ。

## 原則

| 原則 | 意味 |
|------|------|
| 誰も決めていないことは書かない | 合意・計測・コード根拠のない主張は未決事項へ |
| 画面・機能でスコープを切る | 実装詳細は Data Flow / API Design に書き、スコープは「ユーザーから見て何が変わるか」 |
| なぜを構造で表す | 親 bullet = what、子 bullet = why（3 段まで） |
| 数値は実測根拠と併記 | 「余裕」「おおむね」を避け、本番計測・ローカル実測・スペック参照で裏取り |
| YAGNI | 将来分割や先回り制約は書かない、必要になった時点で別 issue |

## 共通ヘッダー

```markdown
# Design Doc: {タイトル}

Status: {draft | proposed | approved | archived}
Created: YYYY-MM-DD
Updated: YYYY-MM-DD   # 必要時のみ
Issue: {Issue URL}    # あれば
PRD: {PRD URL}        # あれば
```

分割型は `summary.md` に集約、各ファイルは「関連: summary.md」のみ。

## 文章構造と図表の型

| タイプ | 形式 / カラム |
|---|---|
| 論理関係（〜だから〜） | 段落（接続詞つき短文） |
| 並列列挙 | bullet（3 段まで） |
| 処理フロー | mermaid sequenceDiagram（User / API / DB の 3〜4 actor） |
| テーブル設計 | `カラム名 / 型 / NULL / デフォルト / 説明` の 5 列 |
| インデックス | bullet（主キー→複合→UNIQUE の順、各に「なぜ」1 行） |
| エンドポイント | `メソッド / パス / 説明`（必要なら「変更内容」列） |
| 代替案 | `方針 / 採否 / 理由` の 3 列、採用 ○ / 却下 × |
| リリース手順 | 番号付きリスト（各に「コマンド」「操作者」「戻し方」） |
| 未決事項 | `# / 決定タイミング / 項目 / 決定方法` の 4 列 |
| リグレッション観点 | `観点 / 検証方法` の 2 列（PRD 参照列は付けない） |

ASCII 罫線のアーキ図は NG、mermaid flowchart を使う。

## アンチパターン（AI 臭・形式崩れ）

検出したら潰す。ドラフト→最終稿で必ずチェック。

| パターン | 例 | 対処 |
|---|---|---|
| テンプレ見出しラベル残骸 | 「- 機能の概要と目的」「- 解決しようとしている問題」が bullet として残る | ラベル削除、本文だけ残す |
| 丁寧な説明口調 | 「〜という問合せ形に合わせるため」 | 結論→理由の順に短文化 |
| 箇条書き多層ネスト | 3 階層以上、子が親の言い換え | 文章化 or bullet 3 段まで |
| 過剰な定量補足 | 「15 秒は実質的な上限値」「60 秒の 25% 程度で余裕」 | 「処理時間 ≒ 15 秒、タイムアウト 60 秒に余裕」で十分 |
| 同内容の言い換え反復 | 同じ注意を 2 箇所で繰り返す | 1 箇所に集約 |
| 網羅列挙 | 「原因は 3 つに限られる」 | 重要なものだけ、網羅を目指さない |
| 敬体・常体の混在 | 「〜します」と「〜する」の混在 | 常体に統一 |
| 実装状況マーカー | ✅ 🚧 ❌ で進捗を書く | Status ヘッダー + 本文で完結させる |
| 番号付き見出し | `### 1. 概要 / ### 2. アーキテクチャ` | `##` / `###` で十分。番号は崩れる |
| 英語版併記 | 同一内容を日英で書く | 日本語のみ。必要なら別ファイル |
| 自己参照の多用 | 「本 DesignDoc は〜」 | 地の文で書く |
| PR 単語の混入 | 「実装 PR で決める」「別 PR 対応」 | DesignDoc は PR を知らない。「実装時に決める」等に置換 |
| 重い造語見出し | 「〜の軸分離」「〜の運用制限見直し」 | 助詞を入れて自然に |
| ゼロ事実の明記 | 「コードに上限バリデーションはない」 | 存在側を書く（「運用ガイド側で制限」等） |
| PRD 対応観点カラム | 本文の表に「PRD 対応観点」「PRD 要求」列を追加 | PRD 参照は Overview / Appendix のみ |
| 実装詳細の貼り付け | 関数呼び出し順序、具体ファイルパス、CSS プロパティ名 | 「何をしたいか」まで。詳細は実装 PR |
| SQL の CREATE TABLE 全文 | DDL をそのまま貼る | テーブル名 + 5 列表 + インデックス bullet のみ |
| 過去経緯の混入 | 「当初は〜」「以前は〜」「元々〜」 | 削除。比較は代替案表に並列で |
| ドラフト中の造語残存 | 案 A / 案 B-lite / 第 N 世代 | 本文では実装名 or 正式名、造語は代替案表内のみ |
| PRD/DesignDoc 混同 | タイトルが `# PRD:` で配置が `DesignDocs/` | タイトルと配置を合わせる |
| 削除残骸文 | 文脈から浮いた一文が単独で残る | ドラフト→最終稿で前後文脈を読み直し、孤立文を削除 |
| WHY 欠落の制約 | 「分割禁止」「単一 Tx に集約」等の制約だけ書いて理由なし | 制約には「なぜそうするか」を 1 行添える（守らないと何が壊れるか） |
| 将来形の条件欠落 | 「将来分割する場合は再設計」とだけ書く | 「どういう状況で必要になるか」を併記。書けないなら YAGNI として削除 |
| 指示語・参照対象の曖昧化 | 「対応する行」「ロックした行」が複数テーブル間で曖昧 | テーブル名・列名で参照先を一意に特定（「`A.x` でロックした行に対応する `B` の行」） |
| 重要操作の省略 | 直前に方式名を書いたから本文で動詞を省く | 重要動詞（シャッフル / ロック / 削除等）は省略せず再掲 |
| 旧方式前提の概念残存 | 新方式で不要になる概念（リトライ上限 / フォールバック等）を旧方式の感覚で残す | 新方式で何が不要になるかを棚卸ししてから書く |
| DB 方言の混同 | `FOR UPDATE OF table_name SKIP LOCKED` を MySQL で書く（PostgreSQL 構文） | 採用 DB の構文で書く。方言差は SQL 節に「採用構文」を明示 |
| 既出議論との不整合 | 朝会・チャットで出た方針を反映せず、独自案を書く | 書く前に直近の議論ログを確認、既出案は代替案表に並べて採否を示す |

## スコープの切り方

- 対象 / 除外は「ユーザーから見て何が変わるか」の範囲で書く
- 実装上必須でも UI/挙動変化がないものは Data Flow / API Design 側に書く
- 同カテゴリの既存 DesignDoc を 1-2 本読んでから書き、見出し・スコープ表記・セクション名を揃える
- 紐付け編集制約・冪等性ガード・将来分割トリガーなど誰も要望していない先回り設計は書かない

## 専門用語の置換

読者によっては初見の用語を日本語化。

| 専門用語 | 置換案 |
|---|---|
| Fisher-Yates | ランダムな順 |
| Parallel Change | 段階移行 |
| O(1) | 件数に依存しない |
| 冪等性ガード | 二重実行防止 |
| 同期ブロッキング | 完了まで応答を返さない |
| スループット | 書き込み速度 |
| 整合性担保 | 整合性を保つ |
| 便益が上回る | 効果に見合わない |

残す: `p95` / SQL 構文（`SKIP LOCKED` / `FOR UPDATE` / `AUTO_INCREMENT`）/ クラウド・OSS の固有名詞。

## 想定 SQL の書き方

コード全文は貼らない。意図のみ bullet で。

| 項目 | 書く内容 |
|---|---|
| 対象 | テーブル・範囲 |
| 条件 | WHERE 節の論理条件 |
| 並び順 | ORDER BY の意味 |
| 件数 | LIMIT |
| ロック | FOR UPDATE / SKIP LOCKED 等の挙動 |

## 影響調査と裏取り

「既存に〜がある」「既に〜実装されている」の前提条件は書く前に必ず裏取り。

| 前提の種類 | 裏取り方法 |
|---|---|
| 既存カラム・テーブル | grep / シンボル検索ツール |
| 関数・型 | シンボル検索ツール / grep |
| 既存 PR のスコープ | リポジトリホスティングのPRビューア |
| 既存 DesignDoc | 最新コミットを全文確認 |
| 他チームの進捗 | 担当者に確認、Issue/Wiki 参照 |

広域分析は探索 agent にコード path・行番号を根拠つきで返させる。

## 数値の書き方

OK: ローカルベンチマーク、本番環境スペック（IaC 参照）、本番データ集計（公開 API・メトリクス）。

NG: 仮定スループットから逆算した近似値、未検証の仮定値。

仮定しか書けないものは未決事項に逃がす。

## 未決事項

「項目 + 決定方法」を対にして書く。決定方法が TBD なら表に載せず削除。

| 本文に書く | 未決に逃がす |
|---|---|
| 合意済みの方針 | 合意前の提案 |
| コード・計測で検証済みの事実 | 仮定・推定 |
| 今回やること | 別 issue でやること |

## 削除候補セクション（軽量テンプレ時）

| セクション | 削除根拠 |
|---|---|
| Goals / Non-Goals | Overview の概要・スコープで代替 |
| Technology Choices | 処理フロー説明内に統合可 |
| Security Considerations | 「既存踏襲」のみなら削除 |
| Testing Strategy 詳細 | 実装 PR のテスト対象 |
| Rollback 表（合意前） | 未決事項へ |
| Why not | テンプレ外。代替案は「代替案表（方針 / 採否 / 理由）」に集約 |
| 前提条件 | テンプレ外。Context に必要分だけ織り込む |

## レビュー対応

各コメントを 3 分類して返す。

| 分類 | 対応 |
|---|---|
| 設計判断変更 | 本文書き直し |
| 説明不足 | 理由・背景を段落で補足 |
| YAGNI 懸念 | 削除、必要なら未決事項へ |

返信は「どこをどう直したか」を 1 行添える。派生した未合意論点は未決事項にリンク。

## セルフチェック

- [ ] スコープは「画面・機能」単位で書かれているか
- [ ] 本文の主張すべてに「誰が決めたか / どう検証したか」の根拠があるか
- [ ] 数値に実測・計測・スペック参照の根拠があるか
- [ ] 代替案の不採用理由が 1 行で説明できるか
- [ ] 未決事項は「項目 + 決定方法」で対になっているか
- [ ] bullet の語尾が揃い、敬体・常体が混在していないか
- [ ] 「PR」「本 DesignDoc」等の自己参照が混入していないか
- [ ] 「〜は存在しない」のようなゼロ事実を書いていないか
- [ ] 関数順序・具体ファイルパス等、実装 PR で決まる詳細を貼っていないか
- [ ] 表記・節構成が同カテゴリの既存 DesignDoc と揃っているか
- [ ] ドラフト中の造語・過去経緯の文が残っていないか
- [ ] 編集中に削除し残した孤立文がないか（前後文脈とつながらない一文）
- [ ] 制約・禁止事項に「なぜ」が 1 行添えられているか
- [ ] 「将来〜の場合」と書いた箇所に発生条件が併記されているか
- [ ] 指示語（「対応する行」「ここ」）の参照先が一意に特定できるか
- [ ] 新方式で不要になった旧概念（リトライ上限・フォールバック等）が残っていないか
- [ ] SQL は採用 DB の方言で書かれているか（`FOR UPDATE` / `SKIP LOCKED` 等の差異）
- [ ] 朝会・チャットの既出議論と本文の方針が整合しているか
