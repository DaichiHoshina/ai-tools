## 技術ドキュメント執筆時の skill 連携

技術系ドキュメント（API設計・バックエンド設計）を書くときは、関連 skill を併用すると用語と構造のブレが減る。

| 書く対象 | 併用する skill | 効果 |
|----------|--------------|------|
| REST/gRPC API 設計、エンドポイント仕様 | `/api-design` | 命名規則・HTTPメソッド・ステータスコード・バージョニングの整合確認 |
| サービス層・データアクセス層・ジョブ設計 | `/backend-dev` | 言語固有パターン（Go/TS等）・エラー処理・トランザクション境界の用語固定 |
| マイクロサービス間連携・モノレポ構成 | `/microservices-monorepo` | bounded context 境界・サービス通信パターン |
| アーキテクチャ判断・DDD | `/clean-architecture-ddd` | レイヤー責務・依存方向・集約境界 |

skill 内のガイドラインを Read してから書き始めると、専門用語を「定義併記なしで多用する」「曖昧な抽象語で逃げる」を防げる。

## 守ること（最優先）

ドキュメントは「読み手の認知負荷を下げる」ための道具。記憶ダンプではなく、判断/行動を助ける手段。すべてのルールはこの目的から派生する。

- 1回読めば理解できる文章を目指す
- 読み手に「で、つまり何？」と思わせない
- 「賢そう」でなく「伝わる」を優先する
- 初見の人が途中で止まらないこと

迷ったときはこの4つに戻る。判断の優先順は「ルール準拠」より「読み手の負荷」。

## 書く前の4問

抽象的な答え不可。答えずに書くと AI 文体になる。

1. 読み手は具体的に誰か（×「エンジニア」○「新joinのsenior backend」「レビュワー Ayano さん」）
2. 読後に何を決めさせる/やらせるか（×「理解させる」○「approveを決めさせる」「migrationを実行させる」）
3. 主張を裏付ける数字 or 事例（×「大幅改善」○「p99 1.2s→320ms (-73%)」）
4. 「なぜ？」と聞かれたら（×「〜が必須」○「localStorageにtoken置かない、XSSで抜かれる」）

## 守る指針（既存ルールと重複しない核）

軸1以降の詳細ルールは、ここを起点に派生する。

1. **用語を並べるだけで説明しない**: 一般化された専門用語の羅列は内容ゼロ。「可観測性」「疎結合」「スケーラビリティ」を並べても何も伝わらない。1用語ごとに「何のために・どう動くか」を1文添える
2. **暗黙知を前提にしない**: 「いつもの」「ご存知の通り」「言うまでもなく」を使ったら、その内容を1文で書く。読み手の前提知識を疑う
3. **次の疑問を先回り**: 1段落書いたら「読み手が次に何を聞きたいか」を想像し、その答えを次段落に置く。質問を放置しない
4. **見出しだけで流れが分かる**: 目次を読むだけで「全体で何が言いたいか」が伝わる構成。見出しはラベル（「アーキテクチャ」）でなく主張（「読み書きを分離して負荷を分散する」）
5. **段落に役割を持たせる**: 各段落は「背景」「理由」「具体例」「結論」「注意点」のどれかに該当させる。役割が曖昧な段落は削除 or 分割
6. **抽象説明の直後に具体例**: 抽象名詞や原則を書いたら、次の1-2文で具体例（数値・事例・コード断片）を置く。抽象だけで段落を終わらせない
7. **「賢そう」より「伝わる」**: 難しい言い回し・凝った修辞より、平易で短い文を選ぶ。読み手は実務判断のために読む、書き手の知識をひけらかすために読むのではない

## 避けるパターン（メタ的なAI臭）

単語レベルのNGは「軸4 内容」で扱う。ここは**段落・文章全体の構造**として避けるべき型。

| パターン | 症状 | 対処 |
|----------|------|------|
| 抽象的なポエム | 名詞だけで構成、絵が浮かばない | 具体的な状況・行動・数値に置換 |
| 用語の羅列 | 専門用語が並ぶが各々の関係が不明 | 関係を1文で繋ぐか、1つに絞る |
| 過剰に丁寧な表現 | 「いただきますよう〜」「〜していただければ〜」 | 直接的な動詞文に短縮 |
| 余分な形容詞 | 「強力な」「革新的な」「柔軟な」が情報を持たない | 削除 or 数値/事例に置換 |
| 同義反復 | 同じ内容を言い換えて繰り返す | 1箇所に集約 |
| 中身のないまとめ | 「以上をまとめると重要だ」で何も結論してない | 1文の結論を書くか、まとめ自体を削除 |

## 段落レベル Before/After

単語・文レベルの例は「軸4」にある。ここは**段落全体の変換**を示す。AI 臭が抜けないとき、まずこの形に書き直す。

### ❌ AI調（抽象ポエム + 用語羅列）

> 可観測性向上を目的として疎結合なイベント駆動アーキテクチャを導入し、スケーラビリティを担保する。これにより、システム全体の柔軟性と保守性が大幅に改善される。

問題点:
- 抽象名詞（可観測性・疎結合・スケーラビリティ・柔軟性・保守性）の羅列で、絵が浮かばない
- 「何が起きていて、何を変えるか」が書かれていない
- 「大幅に改善」の根拠ゼロ
- 読み手は「で、つまり何？」となる

### ⭕ 自然な実務文（具体状況 → 具体解決）

> 注文処理が重くなったときに、一部の処理だけ遅延できるようにしたい。そのため、メール送信や通知処理を別の非同期処理に分離する。本処理（在庫引当・決済）の応答時間を p95 500ms 以内に保つことが目的。

違い:
- 抽象名詞ゼロ、すべて具体的な処理名で書かれている
- 「何が困っていて」「どう解決するか」が時系列で読める
- 数値目標（p95 500ms）で「大幅に改善」を置き換え
- 読み手の頭に絵が浮かぶ

抽象→具体への変換は機械的にできる: 抽象名詞を「いま起きている具体的な事象」「これから取る具体的な行動」に分解する。

## 媒体ごとに構造を変える（短文=PREP、長文=TL;DR+セクション化）

### 短文（issue/PR本体/コメント/Jira/Slack/Notionコメント）

**PREP 3点構造**:
1. 結論: 読み手に何を判断/実行させるか（1行）
2. 理由: 現象/影響/原因
3. 次アクション: 担当/期限/不明点

- 本文400字前後（30秒で読める）、title 80字前後
- 詳細ログ・スタックトレースは `<details>` 折りたたみ or 別ファイルリンク
- 投稿前self-check: ①1行目で判断/実行内容が言えるか ②H3 3個以上 or 1画面超は長すぎ ③結論/理由/次アクション揃ってるか ④評価語に根拠1文あるか

### 長文（Design Doc/PRD/RCA/Notionページ）

- TL;DR冒頭1-3文で結論。「本稿では〜について述べる」系導入は削除
- 800-1500字/ページ、2000字超なら分割
- 1セクション = 地の文3-5割 + 箇条書き/表5-7割
- 箇条書きは3-5項目。7超は表 or 段落分割
- コードブロックは5行以内、超過時は意図1文補足
- 箇条書きだけだと「何故列挙か」が消える → 前後に1-3文の地の文

### PR description

4セクション固定: **Why** / **What changed**（数字・事例）/ **Testing** / **Review focus**

### Design Doc 12セクションテンプレ

1. Overview (PRDリンク) / 2. Goals & Non-Goals / 3. Background (Why) / 4. High-Level Design (図) / 5. Detailed Design (5.1 データモデル, 5.2 API, 5.3 処理フロー) / 6. Alternatives / 7. Trade-offs / 8. Failure Handling / 9. Migration Plan (Expand/Migrate/Contract) / 10. Rollback Strategy / 11. Observability / 12. Open Questions

軽量版（5節）: Overview / Data Flow / Data Schema / API Design / Appendix

## Design Doc は Why/比較/失敗/移行/責務/変更耐性の6つを必ず書く

1. **Why書く**: 技術だけでなくPRD/要件との接続（×「新テーブル作る」○「O(1)抽選のため新テーブル作る」）
2. **比較とトレードオフ**: Alternatives セクション必須、最低2案比較。設計は正解でなく選択
3. **変更耐性**: 「今動く」でなく「変更しやすいか」を書く
4. **責務境界**: service/module間の役割を明確化（DDD的、曖昧だと事故る）
5. **失敗ケース**: 在庫不足・API失敗・二重実行・冪等性など本番で死ぬポイント
6. **移行戦略**: DB変更は Expand → Migrate → Contract の3段で書く

**レベル高い書き方**: 数字で語る（O(n)→O(1)）/ 図で説明（シーケンス・ER・アーキ）/ 制約明示（MySQL 8.0、READ COMMITTED）

## chat と document で文体を分ける（document は常体・主語必須）

| 場面 | 方針 |
|------|------|
| Chat応答（genshijin） | 体言止め可、助詞省略可、結論先出し、短く速く |
| ヒト向けdocument | 文として完結（〜する/〜した）、主語明示、地の文と箇条書き混在、常体統一 |

**document基本ルール**:
- 主語必須（「担当者は〜」「ユーザーは〜」）
- 指示語禁止（「これ」「それ」「上記」「前述」→ 具体名）
- 略語禁止（初出はフルスペル）
- 「適宜」「状況に応じて」禁止 → IF-THEN形式で明文化
- 日付 YYYY/MM/DD、英数字半角

## AI臭を消す3変換（抽象→数字、評価→根拠、難語→定義併記）

### (a) 抽象語 → 数字/事例

| NG | OK |
|----|----|
| レスポンス速度を大幅改善 | p99 1.2s → 320ms (-73%)。Redis cache + N+1削除 |
| エラー処理を強化 | 5xx日次120件 → 8件。`context.WithTimeout`を全handlerに追加 |
| 効率的にドキュ作成可能 | Notionへmd直接投稿できる。手動コピペ廃止が目的 |

### (b) 評価語 → 根拠1文併記

「適切な/最適な/最適化/効率化/改善/重要/必須/推奨/一般的に/通常/基本的に/強化/向上/活用」を書くなら直後に**1文で根拠**（なぜ・事例・数字）。

| NG | OK |
|----|----|
| セッショントークンの適切な管理が必須 | session tokenはcookie `httpOnly`に。localStorageはXSS経由で盗まれ、tokenにexpiryしかないため全アカウント乗っ取り可 |

### (c) 難語 → 削除でなく定義併記

初出で1文の定義を併記。専門用語を削ると正確さを失う。

| NG | OK |
|----|----|
| idempotencyを担保 | idempotency（同リクエストを2回叩いても結果不変）を担保。retryで二重注文発生しない |
| CQRSで整合性 | CQRS（読み書きを別モデル分離）で整合性。readはreplicaから返せてscale可 |

### NG辞書（削除対象）

**AI定型語**: 効果的に / 効率的に / シームレスに / 直感的に / 革新的な / 素晴らしい / 強力な / より良い / 〜を実現します / 〜を提供します / 〜を可能にします / 〜することができます / ご紹介します / ご覧ください / 〜いただけます / ユーザーエクスペリエンス / ユーザビリティ向上 / まず〜しましょう / 重要なポイント / 注目すべき点 / 本機能は / 本ドキュメントは / 本記事では / 本稿では〜について述べる

**硬い文語（柔らかく）**: 〜である / 〜であった / 〜されている / 〜となる / 〜に基づき / 〜に関して / 当該 / 当方 / 貴殿 / 以下に示す

## Design Doc で潰すべきアンチパターン（30種）

ドラフト→最終稿で潰す:

| パターン | 対処 |
|----------|------|
| テンプレ見出しラベル残骸（「- 機能の概要と目的」がbulletに） | ラベル削除、本文だけ残す |
| 丁寧な説明口調 | 結論→理由の順に短文化 |
| 箇条書き3階層以上のネスト | 文章化 or 3段まで |
| 過剰な定量補足（「15秒は実質的な上限」「60秒の25%程度で余裕」） | 「処理時間≒15秒、タイムアウト60秒に余裕」で十分 |
| 同内容の言い換え反復 | 1箇所に集約 |
| 網羅列挙（「原因は3つに限られる」） | 重要なものだけ |
| 敬体・常体の混在 | 常体に統一 |
| 実装状況マーカー（✅🚧❌） | Statusヘッダーで完結 |
| 番号付き見出し（「### 1. 概要」） | `##`/`###`で十分 |
| 自己参照（「本DesignDocは〜」） | 地の文で書く |
| PR単語の混入（「実装PRで決める」「別PR対応」） | 「実装時に決める」等に置換 |
| 重い造語見出し（「〜の軸分離」） | 助詞を入れて自然に |
| ゼロ事実の明記（「コードに上限バリデーションはない」） | 存在側を書く |
| 実装詳細の貼り付け（関数順序・ファイルパス・CSSプロパティ） | 「何をしたいか」まで |
| SQLのCREATE TABLE全文 | テーブル名+5列表+インデックスbullet |
| 過去経緯の混入（「当初は〜」「以前は〜」） | 削除。比較は代替案表に |
| ドラフト中の造語残存（案A/案B-lite/第N世代） | 本文は実装名/正式名、造語は代替案表内のみ |
| PRD/DesignDoc混同（タイトルとパスが不一致） | タイトルと配置を合わせる |
| 削除残骸文（文脈から浮いた一文） | 前後文脈読み直して孤立文削除 |
| WHY欠落の制約（「分割禁止」だけ） | 「守らないと何が壊れるか」を1行 |
| 将来形の条件欠落（「将来分割する場合は再設計」） | 発生条件併記、書けないならYAGNI削除 |
| 指示語の参照対象曖昧 | テーブル名・列名で参照先を一意化 |
| 重要操作の省略（直前で方式書いたから動詞省略） | シャッフル/ロック/削除等は省略せず再掲 |
| 旧方式前提の概念残存（リトライ上限/フォールバック） | 新方式で不要になる概念を棚卸し |
| DB方言の混同（`FOR UPDATE OF table` をMySQLで） | 採用DBの構文で書く |
| 既出議論との不整合 | 朝会・チャットの方針を反映、既出案は代替案表に |

## 図表は型を固定する（mermaid + 5列テーブル）

| タイプ | 形式 |
|--------|------|
| 論理関係（〜だから〜） | 段落（接続詞つき短文） |
| 並列列挙 | bullet（3段まで） |
| 処理フロー | mermaid sequenceDiagram（User/API/DBの3-4 actor） |
| アーキ図 | mermaid flowchart（ASCII罫線はNG） |
| テーブル設計 | カラム名/型/NULL/デフォルト/説明 の5列 |
| インデックス | bullet（主キー→複合→UNIQUEの順、各に「なぜ」1行） |
| エンドポイント | メソッド/パス/説明 |
| 代替案 | 方針/採否/理由 の3列、採用○/却下× |
| リリース手順 | 番号付きリスト（コマンド/操作者/戻し方） |
| 未決事項 | #/決定タイミング/項目/決定方法 の4列 |

想定SQLは全文貼らない。意図のみbullet（対象/条件/並び順/件数/ロック）。

## 書き直しは Phase 1-8 で進む（方針→清書→補強→突合→削除→復元→lint→転換）

長期レビューを経たDesign Doc/PRDの典型フェーズ。

| Phase | 内容 |
|-------|------|
| 1 方針決定・ドラフト | PRD/過去議論/朝会ログを全読してから着手 |
| 2 全面清書 | テンプレ準拠、経緯記述/案A・B造語/網羅志向を排除 |
| 3 自レビュー補強 | Security / API Design / Risks / Performance / Logging / Mermaid図 / 数値根拠の出典を追加 |
| 4 コードベース突合 | migration番号・カラム配置・型・APIパス・前提条件を実装と一致 |
| 5 レビュー反映削除 | テンプレ外/冗長/言い換え反復/自己参照/PR単語を削る |
| 6 削りすぎ復元 | 文脈通らない箇所を再注入、削減と保持を交渉 |
| 7 文体統一・lint | 体言止め/敬体常体/単位（万・億・ms・MB）/textlint（助詞重複/コロン終止/誇張） |
| 8 大方向転換 | 旧記述を残さず一気に新方針へ、変更履歴・経緯セクションも一掃 |

**回数見積もり**: 軽量テンプレ準拠ドラフト 3-5 commits / フルテンプレ 5-10 / 自由形式 20+ / 方針未確定でフル着手 50+。初稿でテンプレ準拠すると往復激減。

**レビュー応答の対応表**:

| 指摘 | 対応 | Phase |
|------|------|-------|
| 「いらない」「これは何？」 | 即削除 | 5 |
| 「もっと詳しく」「分かりにくい」 | 段落で補強 | 3 |
| 「実装と違う」 | コード再確認 | 4 |
| 「方針が違う」 | 過去議論を辿って合意案に戻す | 4 |
| 「テンプレに無い」 | 該当セクション削除 | 5 |
| 「文体混在」「単位揃ってない」 | 一括置換 | 7 |

## レビューは内容→文→構造の3段で別ループ

1回で全部見ると観点が混ざって指摘が鈍る。**内容→文→構造の順で抽象度が変わる**ので段1→2→3が効率的。

| 段 | 観点 | 対象 |
|----|------|------|
| 1 内容整合性 | ファイルパス/構造体フィールド/DB NOT NULL/PRD受け入れ条件の対応抜け/Phase計画の依存/Why-not妥当性 | 既存コードgrep |
| 2 文章規定 | 評価語の根拠/助詞重複/だ・である調/AI listing pattern/抽象語/読後アクション末尾明示 | NG辞書 + textlint |
| 3 読みやすさ | 情報密度/表vs段落バランス/優先度/図解統一/forward referenceのアンカー化/新規読者目線 | AI目視 |

タイミング: 段1は `/design-doc` 出力後PR起票前、段2はPR起票後CI通過後、段3は最後に「初見読者が読めるか」確認。

| 種別 | 保存先 |

| ADR / DesignDocs / PRD / ProductSpec / TechnicalSpec | GitHub（コードと一緒にバージョン管理） |

## Design Doc vs 詳細設計書

| 観点 | Design Doc | 詳細設計書 |
|------|-----------|-----------|
| 焦点 | Why / 設計判断 / トレードオフ | How / 実装仕様 |
| 読み手 | レビュワー / 将来の自分 | 実装者 |
| 価値 | 議論の土台 | 実装の手順書 |

DDD / マイクロサービスではDesign Doc力が支配的（設計判断の積み重ねが品質を決める）。

## Notion固有ルール

- 階層3階層まで、トップカテゴリ5-8個
- 命名: カテゴリ=名詞 / ナレッジ=「[カテゴリ] 具体内容」 / 障害=「[YYYY/MM/DD] 概要」 / ADR=「ADR-NNN: 決定事項」
- 静的ページよりデータベース優先（タグはRelationで別DB化、フリーテキスト禁止）
- H2大見出し/H3小見出し（H1は使わない）、色2色まで、赤字は絶対忘れてはいけない操作のみ、トグル入れ子禁止
- 1ナレッジ=1ページ（複数トピック混ぜない）
- 冒頭にコールアウトで概要・結論
- ページ冒頭にキーワード列挙（全文検索対策）、同義語・別名も含める（「認証」「ログイン」「auth」）
- 更新 > 新規作成、Archived削除しない（過去経緯を辿れる）

## 出力前セルフチェック6項目（5項目以上で合格）

```
[ ] TL;DR/結論が冒頭1-3文（「本稿では〜」削除済）
[ ] 「必須」「推奨」「重要」に1文の根拠併記
[ ] 初出専門用語に定義併記
[ ] 抽象語（改善/最適化/効率化）の代わりに数字 or 事例
[ ] 読み手の次アクション（approve/実行/質問）が末尾明示
[ ] 4問（読み手/ゴール/数字/なぜ）に答えてる
```

未達は修正してから出力。投稿/commit/push前にAI自身が確認。

## Design Doc セルフチェック18項目

```
[ ] スコープは「画面・機能」単位
[ ] 全主張に「誰が決めたか / どう検証したか」の根拠
[ ] 数値に実測・計測・スペック参照の根拠
[ ] 代替案の不採用理由が1行で説明できる
[ ] 未決事項は「項目+決定方法」で対
[ ] bullet語尾統一、敬体・常体混在なし
[ ] 「PR」「本DesignDoc」等の自己参照なし
[ ] 「〜は存在しない」のゼロ事実なし
[ ] 関数順序・具体ファイルパス等の実装詳細なし
[ ] 表記・節構成が同カテゴリ既存DesignDocと揃う
[ ] ドラフト中の造語・過去経緯文なし
[ ] 編集中の孤立文なし
[ ] 制約・禁止事項に「なぜ」1行
[ ] 「将来〜の場合」に発生条件併記
[ ] 指示語の参照先が一意特定可
[ ] 旧概念（リトライ上限・フォールバック等）の残存なし
[ ] SQLが採用DBの方言
[ ] 既出議論と本文方針が整合
```

## ドキュメント種別は PRD→ADR/DesignDoc→ProductSpec/TechnicalSpec の流れ

技術ドキュメントは6種別に分かれ、依存関係がある。1つのトピックを書くとき「どの種別か」を最初に決めると、書く内容が自動で絞られる。

```
PRD（何を作るか・なぜ作るか）
  ↓ 受けて
ADR（技術選定の判断記録） + Design Doc（実装方針）
  ↓ 設計を元に
ProjectDocs（誰がいつどう進めるか）
  ↓ リリース後
ProductSpec（プロダクト仕様・ユーザー視点）+ TechnicalSpec（技術仕様・実装後の現状）
```

| 種別 | 焦点 | 主読者 | 更新タイミング |
|------|------|-------|----------------|
| PRD | What / Why（ビジネス要件） | PM / Eng / Design | 機能着手前 |
| ADR | 技術判断の理由（1判断1ファイル） | Eng（過去判断追跡） | 不変（決定後はsupersededで上書き） |
| Design Doc | How（実装方針・トレードオフ） | レビュワー / 将来の自分 | 設計時、approvedで凍結 |
| ProjectDocs | When/Who/How（進行管理） | PJM / Eng | 進行中 |
| ProductSpec | リリース済プロダクト挙動 | サポート / QA / 新join | 機能変更で更新 |
| TechnicalSpec | 実装済アーキ・実測値・現状仕様 | Eng（既存改修時） | 実装変更で更新 |

**Bounded Context 別フォルダ命名**: ドメイン駆動設計を採用する場合、`{domain}_context/` で区切る（例: `payment_context/` `user_context/` `order_context/`）。ドメイン跨ぎは `general/` または `cross_context/`。

**Design Doc と TechnicalSpec の分離**: Design Docは「設計時の意図」をスナップショット凍結（実装が変わってもDesign Docは更新しない）。TechnicalSpecは「現状の実装」を反映（実装変更で同期）。両者の差分が「設計→実装で何が変わったか」の履歴になる。

## ADR は1テーマ1Decision で書く（Context/Discussion/Decision/Consequences）

`{topic}` 1テーマ・`Decision` 1つの原則。複数決定を1ADRに混ぜない。

```markdown
# ADR: [タイトル]

Status: [proposed | accepted | rejected | deprecated | superseded by ADR-XXX]
Created: YYYY-MM-DD

## Context（背景）
- 決定したいことの背景・動機
- 現状の課題・制限
- 影響を受けるステークホルダー

## Discussion（議論）
- 検討した代替案のpros/cons
  - 案1: ...
    - Pros: ...
    - Cons: ...
  - 案2: ...
    - Pros: ...
    - Cons: ...

## Decision（決定）
- このADRで決定すること **1つ**
- 決定に至った主な理由

## Consequences（決定の結果）
- 短期: プラス影響 / マイナス影響
- 長期: プラス影響 / マイナス影響

## Compliance（遵守方法）
- どう運用で担保するか（lint / レビュー / hook 等）

## Notes（任意）
- 参照資料・関連ADR・レビュー履歴
```

**注意**: ADRは「過去にこう決めた」記録なので、決定が覆る場合は元のADRを編集せず、新ADRを作って `Status: superseded by ADR-XXX` で上書きする。

## PRD は MoSCoW（Must/Should/Could/Won't）で優先順位を固定する

実装着手前にビジネス要件・ユーザーストーリーを固定する。

```markdown
# [機能名]: PRD

## 概要
機能の全体像を3-5行。誰のどんな体験か。

## 課題と目的
### 課題
- 現状の問題点（箇条書き）
### 目的
- 達成したい目的

## ユーザーストーリー（MoSCoW優先順位付け）
### Must have（必須）
- [ユーザー]として[実現したいこと]をしたい。なぜなら[理由]
### Should have（重要だが必須でない）
### Could have（あれば望ましい）
### Won't have（今回はやらない）

## 機能仕様
- 基本機能と詳細仕様
- 各機能の制約条件

## 非機能要件
- パフォーマンス（p95レイテンシ・スループット目標）
- セキュリティ（認証・認可・データ取扱）
- 可用性・スケーラビリティ

## 技術的考慮事項
- 制約となる既存システム・API
- 他チーム依存

## UX/UI仕様
- 画面遷移・主要画面のワイヤー

## テスト計画
- 受け入れ基準・回帰観点

## 成功指標（KPI）
- 数値で測れる成功定義

## スケジュールと依存関係
- マイルストーン・他チーム依存

## 関連ドキュメント
- 関連PRD / ADR / Design Doc へのリンク
```

**MoSCoW運用**: Must/Should/Could/Won't の4段で優先順位を明示。「全部Must」は優先順位放棄なのでNG。Won'tを書くと「やらないこと」が合意され、後でスコープ拡大を防げる。

## 受け入れ基準は EARS（WHEN/IF/WHERE + THEN）で書く

受け入れ基準・バリデーション仕様は **EARS形式** で書く。自然言語の曖昧さを排除して、テストケースが機械的に書ける。

**基本パターン**:

```
WHEN [イベント] THEN システムは [応答]
IF [前提条件] THEN システムは [応答]
WHERE [配置/状態] THEN システムは [応答]
WHILE [継続的状態] THEN システムは [応答]
```

**例**:

```
WHEN ユーザーが「購入」ボタンを押下した場合
THEN システムは決済処理を開始する

IF 在庫数が0の場合
THEN システムは「在庫切れ」エラーを返す

WHERE ソート順がNULLの場合
THEN システムは該当アイテムを最後に表示する
```

### バリデーション網羅観点（必須チェックリスト）

入力項目ごとに以下を漏れなく書く:

- [ ] **データ型**: 整数以外・文字列以外・真偽値以外
- [ ] **範囲**: 最小値未満・最大値超過・文字数超過
- [ ] **必須/任意**: 必須項目が未入力・空文字列・NULL
- [ ] **形式**: メールアドレス・電話番号・URL・日時形式
- [ ] **NULL**: NULL許可項目の動作 / NULL不許可項目のエラー
- [ ] **関連性**: 開始日時 > 終了日時 / 最小値 > 最大値
- [ ] **重複**: 既登録・ユニーク制約違反
- [ ] **存在**: 指定IDが存在しない・削除済み

「正常系のみ書いて出す」がよくあるバグ温床。8観点を機械的にチェック。

## textlint で文体崩れを機械検出する（文末「。」/絶対表現/助詞重複/だ・である調混在/AI listing）

人間目視で見逃しやすい文体崩れは textlint（`@textlint-ja/ai-writing` + `textlint-rule-preset-ja-technical-writing`）で機械検出する。CI conclusion が success でも annotation が付くため、PR作成前にローカル実行。

### 頻出 NG パターン

| 違反 | NG 例 | OK 例 |
|------|-------|-------|
| 文末「。」なし（見出し的に名詞句で終わる） | `含むもの` | `本設計では以下を扱う。` |
| 絶対表現 | `完全に保証する` / `絶対に〜` / `必ず〜される` | `確実に届ける` / `〜される` |
| 助詞重複（一文に同じ助詞2回以上） | `処理が決済を含むため整合性が壊れる`（「が」「が」） | 文を分ける or 一方を「は」へ |
| だ・である調混入（常体ベースに「である」混在） | `〜は大きいからである。` | `〜は大きい。` / `〜のためだ。` |
| AI listing pattern（リスト全項目が「**強調**: 説明」形式） | `- **高**: 設計を確定する前に〜`<br>`- **中**: 設計は変わらないが〜` | 表に変換 or 地の文に展開 |

**事実宣言の絶対表現は OK**: 「最終的に必ず1対1にマッピングされる」のような数学的事実宣言は絶対表現でも通す。禁止対象は「主観的に必ず守られる」のような根拠なき言い切り。

### ローカル実行

```sh
npx textlint -f pretty-error <path>.md
npx markdownlint-cli2 <path>.md
```

### CI annotation 確認（GitHub）

```sh
# 該当 SHA の check-runs を一覧
gh api repos/{owner}/{repo}/commits/<SHA>/check-runs \
  --jq '.check_runs[] | {name, conclusion, count: .output.annotations_count}'

# annotations_count > 0 の job の詳細
gh api repos/{owner}/{repo}/check-runs/<ID>/annotations
```

`.github/workflows/` パスへの annotation は CI infra 由来（Node deprecation 等）で本文無関係なら無視可。

## 設計/計画/実行のフェーズ境界を越境しない

ドキュメント駆動開発では「設計フェーズで実装を始める」「計画フェーズで設計を変える」が典型的な事故ポイント。フェーズの責務を越境しない。

| フェーズ | 許可 | 禁止 | よくある違反 |
|---------|------|------|--------------|
| 設計（Design Doc / ADR） | 要件分析・代替案比較・設計判断 | 実装・テスト作成・タスク分解 | 設計中にコードを書き始める |
| 計画（タスク分解 / ProjectDocs） | タスク定義・依存整理・リスク評価 | 実装・設計変更・新機能追加 | 計画書を書きながら実装に手を出す |
| 実行（実装 / テスト） | 実装・テスト・ファイル操作 | 設計変更・スコープ拡大・新仕様追加 | 実装中に「ついでに」リファクタやスコープ追加 |

**越境を検出する質問**: 「いま書いているのは、このフェーズの成果物か？」「これは次フェーズで書くべきものでないか？」答えに迷う時点で越境している。

## 全体マップ（各ルールがどう認知負荷を下げるか）

このドキュメントの全ルールは、最優先「読み手の認知負荷を下げる」から派生している。下の表は10ルールがそれぞれ「読み手の何の労力を肩代わりするか」を1行で示す。ルールに従うこと自体が目的ではなく、**読み手が払うはずだった負荷を書き手が肩代わりする**ためにルールがある、と理解する。

| ルール | 中身 | 認知負荷をどう下げるか |
|--------|------|---------------------|
| 4問 | 読み手/行動/数字/なぜ | 「誰のためか」を最初に絞り、読み手が自分事として読める |
| 媒体別構造 | 短文=PREP 400字、長文=TL;DR+12セクション、PR=4セクション | 形が決まっていれば読み手は内容に集中できる |
| 文体分離 | document は常体・主語必須・指示語禁止 | 主語と指示が明確だと読み返し不要 |
| 3変換 | 抽象→数字、評価→根拠、難語→定義 | 抽象語の解釈労力を書き手が引き受ける |
| 書き直しPhase | 方針→清書→補強→突合→削除→復元→lint→転換 | 各フェーズで観点を絞ることで書き手の判断も読み手の理解も簡単になる |
| レビュー3段 | 内容→文→構造 | 段階ごとに観点を分けると指摘が重ならず読み手が直しやすい |
| 種別マップ | PRD→ADR/DesignDoc→ProductSpec/TechnicalSpec | 種別が分かれば「何を期待して読めばいいか」が即決まる |
| EARS | WHEN/IF/WHERE + THEN + 8観点 | 自然言語の曖昧さを排して読み手にテストケースが見える |
| textlint | 文末/絶対表現/助詞重複/だ・である調/AI listing | 文体崩れを機械検出して書き手の見落としを救う |
| フェーズ境界 | 設計/計画/実行を越境しない | 文書ごとに「いま読んでいる段階」が明確 |

迷ったら表に戻る前に、まず「これは読み手の負荷をどう下げるか」を自問する。答えられないルール適用は形だけのルール準拠で、結局 AI 調に戻る。
